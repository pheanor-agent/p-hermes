# p-hermes 도메인별 다매체 기술 문서 종합 설계 보고서

## 다매체 출판 전략 및 에르메스 에이전트 시스템 개요

현대의 자율형 인공지능 에이전트 아키텍처는 단순한 질의응답을 넘어선 지속적이고 자율적인 작업 수행 능력을 요구한다. 2026년 2월 미국 오픈소스 AI 생태계의 선두 주자인 Nous Research에 의해 공개된 에르메스(Hermes) 에이전트는 사용자의 개입 없이 독립적으로 학습하고 도구를 활용하며 스킬을 진화시키는 고도의 자율적 인공지능 플랫폼이다. 에르메스는 종래의 단일 세션 기반 챗봇이나 특정 통합 개발 환경(IDE)에 종속된 코딩 보조 도구(Copilot)와는 근본적으로 다른 궤도를 그린다. 이 플랫폼은 로컬 서버나 가상 사설 서버(VPS)에서 영구적으로 구동되며, 과거의 상호작용을 기억하고, 복잡한 문제를 해결한 후 스스로 스킬 문서를 생성하여 동일한 문제에 대한 해결 능력을 영구적으로 내재화하는 폐쇄형 학습 루프(Closed learning loop)를 특징으로 한다.

이러한 복합적이고 다층적인 시스템은 단일한 형태의 기술 문서로는 그 아키텍처의 깊이, 기술적 당위성, 그리고 실제적인 사용 절차를 모두 포괄하여 전달하기 어렵다. 따라서 본 설계 보고서는 p-hermes 배포 범위 내에 존재하는 핵심 운영 도메인(A1~A6)과 배경 환경 도메인(B1~B2)에 대하여, 실무 지침서 역할을 수행하는 위키(Wiki), 기술적 철학과 설계 배경을 전달하는 블로그(Blog), 그리고 핵심 개념의 시각적 압축 전달을 목적으로 하는 슬라이드(Slides)라는 삼원화된 매체 문서 전략을 수립한다.

각각의 문서는 대상 독자의 기술적 숙련도, 정보의 수용 목적, 그리고 열람 환경에 따라 그 어조와 흐름, 레이아웃 및 그래픽 요소가 철저하게 분리되고 최적화되어 설계된다. 위키는 엄격하고 지시적인 어조를 통해 정확한 정보의 전달과 절차적 재현성을 보장하며, 블로그는 설득력 있는 서사를 바탕으로 에르메스 아키텍처가 채택한 설계의 당위성을 논증하고, 슬라이드는 직관적인 공간 배치와 압축된 메시지를 통해 개념적 뼈대를 구축한다.

## 매체별 범용 설계 사양 및 시각적 규격

| 매체 유형 | 핵심 목적 및 타겟 독자 | 최적 분량 | 어조 및 문체 | 논리적 전개 흐름 | 레이아웃 및 그래픽 요소 |
|-----------|----------------------|----------|-------------|-----------------|----------------------|
| **위키 (Wiki)** | 절차적 가이드, API 레퍼런스, 트러블슈팅 (실무 엔지니어 대상) | 3,500자+ | 객관적, 지시적, 간결함 | 전제조건 → 설정 방법 → 단계별 실행 절차 → 검증 및 예외 처리 | 2단 사이드바 탐색 구조. Monospace 폰트 코드 블록. 시퀀스 다이어그램 및 표 형식 활용. |
| **블로그 (Blog)** | 아키텍처 철학, 의사결정 배경 (아키텍트, AI 연구자 대상) | 10,000자+ | 분석적, 해설적, 설득력 있는 비평적 문체 | 기존 시스템의 한계 → 설계 철학 → 심층 메커니즘 분석 → 향후 전망 | 1단 텍스트 중심 구조. 여백 활용한 가독성. 아키텍처 맵, 데이터 흐름도, 인포그래픽 배치. |
| **슬라이드 (Slides)** | 개념적 이해, 온보딩 (신규 합류자 대상) | 8~10장 | 단정적, 압축적, 시각적 직관성 | 후크 → 핵심 개념 3원칙 → 시각적 파이프라인 → 실행 요약 | 그리드 기반 미니멀리즘. 텍스트 최소화, 고대비 타이포그래피. 상태 전이, 계층 구조 인포그래픽. |

**설계 원칙**:
- **교차 링크 최소화**: Wiki/Blog/Slides 간 상호 참조는 과도한 X-Link를 지양. 단일 진입점(README)에서 접근성 보장.
- **분량 기준**: 한국어 자수 기준 적용 (Wiki 3,500자+, Blog 10,000자+).
- **슬라이드 수**: 도메인별 8~10장으로 통일. Guy Kawasaki 10-20-30 Rule 준수.
- **부정대조 패턴 금지**: `~대신`, `~아니라`, `~반면` 등 부정적 대조구 사용 금지. 긍정적 직접 서술.

## A1. Workflow (워크플로우) 도메인 문서 설계

워크플로우 시스템은 에이전트에 대한 작업 요청 수신부터 최종 완료까지의 모든 과정을 추적하고 제어하는 9단계의 상태 기반 작업 파이프라인이다.

### 위키 문서 (3,500자+)

`create-job.sh`를 실행하여 `~/.hermes/workspace/jobs/JOB-XXXX/` 구조의 물리적 폴더를 생성하는 순간부터 시작되는 일련의 절차를 매뉴얼 형태로 기술한다. `request`에서 `done`에 이르는 9단계의 상태 전이를 각각 독립된 하위 섹션으로 분리하여 서술하고, 각 단계에서 요구되는 전제 조건과 산출물을 명시한다. `workflow-gate.sh`의 CLI 출력 예시를 터미널 박스 레이아웃으로 제공하며, `.workflow-state` JSON 파일이 `flock` 명령어를 통해 원자적으로 갱신되는 과정을 시퀀스 다이어그램으로 시각화한다.

### 블로그 문서 (10,000자+)

초기 에이전트 모델들의 불안정성을 분석하며, 무분별한 병렬 작업이 야기하는 상태 충돌(State Collision)의 위험성을 지적한다. p-hermes가 채택한 9단계 철학이 어떻게 조사(investigation), 설계(design), 실행(execution) 단계를 분절시키고, 중간에 리뷰(review)와 승인(approval)이라는 인지적 허들을 배치하여 AI의 환각에 의한 무분별한 코드 파괴를 방어하는지를 아키텍처적 당위성으로 서술한다. 유향 비순환 그래프(DAG) 형태의 플로우차트를 배치하여 상태 머신의 구조를 시각화한다.

### 슬라이드 문서 (8~10장)

9단계 상태 머신의 파이프라인을 핵심 그래픽 요소로 구성한다. 각 슬라이드가 진행될 때마다 파이프라인의 진행률 바가 동적으로 채워지는 효과를 적용한다. 텍스트는 최소화하고, 각 상태 단계로 진입하기 위한 핵심 승인 게이트 조건을 화면 중앙에 볼드체 키워드로 제시한다.

## A2. Spec-Driven Dev (사양서 기반 개발) 도메인 문서 설계

사양서 기반 개발 도메인은 시스템 운영에 있어 사양서(Spec)를 유일한 단일 진실 출처(SSOT)로 강제함으로써 설계 단계부터 검증 단계까지를 견고하게 구조화하는 개발 방법론이다.

### 위키 문서 (3,500자+)

`specs/active/SPEC-XXX.md`를 작성하는 구조적 규칙과 필수 포함 항목을 나열한다. 사용자가 사양서 수정을 우회하여 직접 소스코드를 변경하려 할 때 트리거되는 7개의 검증 스크립트의 에러 메시지를 경고 블록 형태로 시각화한다. 사양서의 생명주기인 5단계 상태 주기를 순차적으로 추적하며, `--run-tests` 플래그를 통한 자동화 검증 파이프라인의 실행 방법을 명령어와 출력값 대비 형태로 나열한다.

### 블로그 문서 (10,000자+)

'구조적 강제(Structural Forcing)' 철학을 중심으로 전통적인 소프트웨어 공학 환경에서 문서와 코드가 분리됨으로써 발생하는 문서 부패(Documentation rot)와 기술 부채의 악순환을 분석한다. 에르메스의 사양서 기반 개발 방법론이 어떻게 모든 코드 변경의 기원을 추적 가능하게 만들고, 코드가 문서를 이탈하는 현상을 원천 차단하는지를 논증한다. 뫼비우스의 띠 형태를 차용한 순환 다이어그램으로 사양서 변경이 코드 갱신과 테스트 자동화로 직결되는 선순환 구조를 시각화한다.

### 슬라이드 문서 (8~10장)

"직접 파일 수정 금지"를 도입부 슬라이드 중앙에 배치한다. 중앙에 위치한 마스터 사양서로부터 코드 컴포넌트, 검증 스크립트, 변경 로그가 방사형으로 뻗어나가는 신경망 맵 레이아웃을 채택한다.

## A3. Content System (콘텐츠 시스템) 도메인 문서 설계

콘텐츠 시스템은 Wiki, Blog, Slides의 콘텐츠를 타겟 독자의 페르소나에 맞춰 자동 생성하고 그 품질을 다층적으로 검증하는 메타 파이프라인이다.

### 위키 문서 (3,500자+)

`persona_generator.py` 및 `tone-adapter.py` 모듈의 조작법과 파라미터 튜닝 가이드를 중심으로 기술한다. `anti-slop-library.json`을 구성하고 금지어 리스트를 업데이트하는 방법을 단계별로 안내한다. 문서 유형에 따라 어조를 D1에서 D5 레벨로 조정하는 척도와, 각 검증 게이트(L1-L4)의 기능 설명을 코드 스니펫과 함께 배치한다.

### 블로그 문서 (10,000자+)

AI 생성 텍스트의 'Slop(무가치한 텍스트 찌꺼기)' 현상을 타파하기 위한 파이프라인의 아키텍처 철학을 다룬다. 5계층 검증 게이트가 어떻게 AI의 환각을 걸러내는지에 대한 방어적 프로그래밍 관점의 서사를 펼친다. L5 심사 모델(Judge Model)이 별도의 LLM 호출을 통해 품질을 심사하는 메커니즘의 우월성을 부각하며, 정제 파이프라인 다이어그램으로 시각화한다.

### 슬라이드 문서 (8~10장)

L1 Schema → L2 Error → L3 Voice → L4 Domain → L5 Judge Model로 이어지는 검증 단계를 스텝 뷰 레이아웃으로 구성한다. 금지어 패턴이 차단되고 긍정 서술이 강제되는 과정을 Before/After 그래픽으로 시각화한다.

### Content System 개선 사항

| 항목 | 개선 내용 |
|------|-----------|
| **Emotion Merger 조건화** | Blog 서사형에만 적용. Wiki 기술 문서에서는 비활성화 |
| **부정대조 패턴 검출** | `anti-slop-library.json`에 `~대신`, `~아니라`, `~반면` 패턴 자동 감지/교정 규칙 추가 |
| **명칭 통일 검증** | `Expression System` → `Content System` 강제 검증 규칙 추가 |
| **L5 비용 최적화** | Wiki는 L1-L4까지만 적용. Blog(긴 분량)에만 L5 Judge Model 추가 호출 |
| **매체별 구조 템플릿** | Wiki: 헤딩 레벨 2-3, Blog: 헤딩 레벨 1-3, Slides: HTML 전용 템플릿 |

## A4. Knowledge System (지식 시스템) 도메인 문서 설계

지식 시스템은 다중 소스에서 파편적으로 발생하는 지식을 수집, 분석 및 점수화하여 계층적 구조의 영구적인 위키 데이터베이스로 관리하는 지식 플랫폼이다.

### 위키 문서 (3,500자+)

지식 시스템의 트리 구조를 탐색하고 갱신 메커니즘을 제어하는 매뉴얼로 구성한다. 세션 이력, JOB 기록, 뉴스 피드 등의 원본 소스가 `~/.hermes/knowledge/wiki/` 하위에 저장되는 과정을 구조적으로 설명한다. LLM 추정을 금지하고 원본 직접 참조만 허용하는 규칙을 경고창으로 삽입한다. `wiki-process-filings.sh`, `build-scores.sh`, `knowledge-sync.sh` 스크립트의 실행 절차를 순서도로 제공한다.

### 블로그 문서 (10,000자+)

일시적인 대화 및 작업 세션 데이터가 영구적인 지식 자산으로 승화되는 '인식론적 지식 설계 철학'을 다룬다. 에르메스 에이전트의 폐쇄형 학습 루프(GEPA) 원리를 지식 시스템과 연계하여 분석하며, '메모리 드리프트' 문제를 방지하기 위한 동적 채점 로직을 논증한다. 3-Tier 점수 체계가 컨텍스트 윈도우 한계를 극복하는 아키텍처적 우아함을 서술한다.

### 슬라이드 문서 (8~10장)

파편화된 원본 데이터 층위에서 고도로 정제된 지식(T1 Wiki)으로 상승하는 3단 피라미드 다이어그램을 메인 레이아웃으로 구성한다. 점수화 알고리즘이 낮은 가치의 데이터를 걸러내고 중요 데이터를 상위 계층으로 밀어올리는 과정을 필터 메타포로 시각화한다.

## A5. Cron/Automation (자동화) 도메인 문서 설계

주기적인 작업의 스케줄링, 이벤트 기반 통신, 자동 실행 인프라를 총괄하는 자동화 도메인이다.

### 위키 문서 (3,500자+)

`~/.hermes/cron/registry.yaml`의 작성 문법과 스케줄러 환경설정에 관한 실무적 레퍼런스를 제공한다. cron 표현식과 `30m`, `every 2h` 같은 확장된 schedule 문법의 활용법을 예시 코드와 테이블로 정리한다. `enabled_toolsets`를 통한 도구 제한 방법과 model 속성의 사용법을 기술하며, `system-common` 라이브러리의 유틸리티 함수들을 백과사전식으로 나열한다.

### 블로그 문서 (10,000자+)

L1 Scheduler, L2 Runtime, L3 Delivery로 이어지는 3층 구조 아키텍처 철학을 해체하고 분석한다. `event.sh` 단일 진입점을 활용한 이벤트 버스 설계가 모듈 간의 결합도를 어떻게 낮추는지에 대해 소프트웨어 공학적 논증을 전개한다. silent-on-success 원칙이 알림의 홍수로부터 사용자의 주의력을 보호하는 철학을 심도 있게 서술한다.

### 슬라이드 문서 (8~10장)

L1에서 L3로 이어지는 3단계 파이프 형태의 인포그래픽을 기본 템플릿으로 적용한다. 파이프라인 종단에서 Telegram, Discord 등 다채널로 갈래를 치는 모습을 시각화한다. L1은 청색, L2는 회색, L3는 녹색으로 배색하여 계층 간 책임을 시각적으로 구분한다.

## A6. Model Routing (모델 라우팅) 도메인 문서 설계

작업 유형별로 최적의 LLM을 선택하고, 교차 검증을 통해 출력 품질을 보장하는 라우팅 시스템이다.

### 위키 문서 (3,500자+)

`model-roles.yaml` 파일의 선언적 문법과 provider 기반 라우팅 규칙을 설명한다. 접두사 무시 규칙의 처리 매커니즘을 예시로 구체화하고, `/model <name>` 명령어를 통한 핫스왑 절차를 안내한다. `catalog.json`의 갱신 절차와 다중 공급자 API 통합 설정 방법을 기술하며, 역할-모델 매핑 테이블을 삽입한다.

### 블로그 문서 (10,000자+)

다중 모델 연동의 전략적 우위와 교차 검증 철학을 주제로 한다. 단일 모델의 환각이 단일 장애점(SPOF)으로 이어지는 문제를 라우팅 시스템이 해결하는 방식을 분석한다. 설계와 검토 단계에서 서로 다른 모델을 교차 할당하는 메커니즘을 인지적 다양성 관점에서 해석하며, Tool Gateway의 통합적 이점을 함께 서술한다.

### 슬라이드 문서 (8~10장)

"하나의 에이전트, 다수의 브레인(One Agent, Multiple Brains)"을 첫 화면에 배치한다. 라우터가 작업 맥락에 따라 모델로 요청을 분기하는 과정을 트랙 전환 애니메이션 메타포로 시각화한다. Tool Gateway의 통합적 이점과 벤더 락인 탈피를 키워드로 요약한다.

## B1. Hermes Agent 개요 도메인 설계

에르메스 에이전트 플랫폼의 통합적 능력과 비전을 소개하는 배경 도메인이다.

### 위키 문서 (3,500자+)

에르메스 생태계의 진입점 역할을 하는 '시작하기(Getting Started)' 가이드로 구성한다. macOS, Linux, Windows WSL2 환경을 포괄하는 설치 명령어를 스니펫 박스로 부각시키며, `hermes setup --portal`을 통한 OAuth 로그인 및 Tool Gateway 활성화 절차를 단계별로 명시한다. `~/.hermes/skills/`를 필두로 한 물리적 파일 트리 구조를 텍스트 다이어그램과 함께 제공한다.

### 블로그 문서 (10,000자+)

자율적 에이전트 패러다임을 개척하는 Nous Research의 비전을 조명하며 거시적 서사를 구축한다. 사용자가 키보드에서 손을 떼더라도 서버에 상주하며 프로젝트를 학습하고 도구를 선택하여 브라우저를 제어하는 에르메스의 영속적 생명력에 대해 철학적 통찰을 제공한다. MLOps를 위한 훈련 데이터 합성 기계이자 강화학습 실험 플랫폼으로 응용되는 공학적 의의를 서술한다.

### 슬라이드 문서 (8~10장)

Nous Portal 통합, 300+ 모델 지원, 다중 챗 플랫폼 연동, 브라우저 제어 능력을 노드 네트워크 다이어그램으로 제시한다. 화면 중앙에 코어 에이전트를 배치하고, 도구, 메모리, 인터페이스를 궤도 위성 형태로 원형 배치하여 구심력과 확장성을 강조한다.

## B2. 아키텍처 도메인 설계

5-Tier 아키텍처와 이벤트 기반 통신을 설계 철학으로 하는 시스템 구조 도메인이다.

### 위키 문서 (3,500자+)

Core, Runtime, Interfaces, Infra, Release로 수직 분리된 5-Tier 구조의 디렉토리 맵과 계층별 책임을 상세히 해설한다. `$HERMES_ROOT` 환경변수 기반 상대 경로 참조 강제 원칙과 심링크 금지 원칙을 대조 표에 명시하며, 파일 시스템에 미치는 영향을 코드 트리 시각화로 전달한다.

### 블로그 문서 (10,000자+)

동기식 결합도를 부수고 이벤트 기반 비동기 통신을 채택한 아키텍처 철학을 논증한다. 스크립트 간 직접 호출이 발생하는 교착 상태 위험성을 지적하고, `event.sh` 단일 진입점 도입의 확장성 관점에서 해설한다. JSONL 이벤트 히스토리의 감사 추적성과 디버깅 편의성을 분석하며, 예측 가능한 소프트웨어 환경 구축의 설계 철학을 서술한다.

### 슬라이드 문서 (8~10장)

5-Tier 아키텍처의 레이어 케이크 구조 다이어그램을 입체적으로 배치한다. 이벤트 버스의 흐름을 점선 애니메이션 광선으로 처리하여 계층 경계면을 관통하게 하고, 모듈 간 느슨한 결합 특성을 시각적으로 유도한다.

## 통합적 문서 생태계 비전

본 설계 보고서는 에르메스 에이전트의 중추를 구성하는 p-hermes 배포 범위 내 핵심 시스템 6개와 배경 도메인 2개에 대한 위키, 블로그, 슬라이드 산출물의 청사진을 제시하였다.

A1~A6 핵심 운영 시스템의 문서들은 작업 상태의 원자적 무결성과 다중 모델 연동을 통한 처리의 효율성을 증명하는 방향으로 어조가 통제된다. B1~B2 배경 도메인 문서들은 MLOps로의 확장성과 플랫폼의 사상을 전파하는 데 역량을 집중한다.

이 설계를 뼈대로 삼아 구현될 다매체 문서 생태계는 실무를 수행하는 엔지니어에게는 절차적 실행 환경을 제공하고, 아키텍트와 연구자에게는 논리적 완결성을 부여하며, 신규 합류자에게는 플랫폼 비전을 직관적으로 전시하는 지식 전달 매트릭스를 구축한다.
