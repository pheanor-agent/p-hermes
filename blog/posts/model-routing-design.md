# 역할 기반 모델 라우팅 설계

> 태그: #architecture #routing
> 읽는 시간: ~3분

## TL;DR
"하나의 모델로 모든 것을 하려는 것은 드라이버가 모든 경주를 이기려는 것과 같다." Hermes는 작업의 단계(Design, Execution, Review 등)마다 가장 적합한 모델을 **역할 기반 라우팅**을 통해 자동으로 배정합니다.

---

## 배경: "만능 모델의 환상"
출시 초기에는 가장 성능이 뛰어난 모델을 모든 단계에 투입했습니다. 
그런데 문제가 생겼습니다.
1. **비용 폭주**: 코드 실행이나 단순 파일 복사에도 가장 비싼 모델을 호출하면 비용이 천문학적으로 증가합니다.
2. **성능 불일치**: 추론 능력이 가장 뛰어난 모델이 항상 코드를 가장 잘 작성하는 것은 아닙니다. (예: 일부 모델은 논리적 설계에는 뛰어나지만, 복잡한 Python 스크립트 작성에는 약함)

---

## 설계 결정: Role-Based Routing

Hermes는 `catalog.json`에서 각 단계에 사용할 모델을 정의합니다.

### 1. Design / Review: 추론형 모델
- **특성**: 논리적 추론, 창의성, 종합 분석 능력이 뛰어남.
- **역할**: 복잡한 아키텍처 설계, 코드 리뷰, 설계서 검증.
- **예시**: High-Reasoning-Model, Cross-Check-Model

### 2. Execution / Coding: 수행형 모델
- **특성**: 프로그래밍 언어 이해도가 높고, 코드 작성 속도가 빠름.
- **역할**: 실제 파일 수정, 테스트 스크립트 작성, 디버깅.
- **예시**: High-Speed-Coding-Model

### 3. Investigation: 범용형 모델
- **특성**: 빠른 응답과 폭넓은 지식.
- **역할**: 파일 검색, 시스템 현황 파악.

---

## 상호 견제 (Cross-Check) 구조
Hermes에서는 **Design 모델과 Review 모델을 다르게 설정**하는 것을 권장합니다. 
- 예를 들어, A 모델이 설계를 하면 B 모델이 그 설계를 검증합니다. 
- 이는 인간의 "타당성 검사"와 동일한 원리로, 설계서 속의 논리적 오류를 훨씬 정확하게 잡아냅니다.
