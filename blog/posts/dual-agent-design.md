# 듀얼 에이전트(Hermes + OpenClaw) 설계

> 태그: #architecture #agents
> 읽는 시간: ~5분

## TL;DR
Hermes 시스템은 한 가지 원칙을 따릅니다: "워커(Worker)는 오직 작업을 수행하고, 오케스트레이터(Orchestrator)는 오직 관리를 수행한다." 이를 위해 **Hermes(워커)**와 **OpenClaw(관리자)**로 시스템을 분리하여, 각 에이전트가 자신의 도메인에서만 활동하도록 설계했습니다.

---

## 배경: "하나의 에이전트에게 모든 것을 맡기다"
초기에는 하나의 에이전트가 모든 것(사용자 요청 처리, 시스템 모니터링, 백업, 지식 관리)을 수행했습니다. 

그러나 다음과 같은 한계에 직면했습니다:
1. **컨텍스트 혼재**: 사용자가 "소설 써줘"라고 요청했는데, 에이전트가 동시에 "디스크 용량 감시"와 "지식 동기화"를 수행하려다 둘 다 잘못 수행하는 현상.
2. **동시성 문제**: 백업 중 컨텍스트가 압축되면, 동시에 요청된 사용자 작업이 데이터 손실을 겪을 수 있음.
3. **책임 소재 불명확**: 오류가 발생했을 때, 사용자가 지시한 명령이 잘못되었는지, 아니면 시스템 관리 로직이 충돌했는지 파악이 어려움.

---

## 설계 결정: 분업의 원칙

Hermes 시스템은 두 개의 에이전트 인스턴스로 나뉩니다.

### 1. Hermes (워커 / Worker)
- **역할**: 사용자의 요청을 받아들이고, 실제 코드를 작성하며, 파일을 수정합니다.
- **도구**: 파일 조작, 코드 실행, 웹 검색, 브라우저 제어.
- **정신**: "나는 결과를 만드는 사람입니다."

### 2. OpenClaw (관리자 / Administrator)
- **역할**: 시스템의 건강 상태를 모니터링하고, 백업을 관리하며, 지식 시스템의 동기화를 제어합니다.
- **도구**: Healthcheck, Security, Blackboard 상태 저장, 세션 DB 관리.
- **정신**: "나는 시스템이 무너지지 않게 하는 사람입니다."

---

## 통신 방식: Blackboard 패턴
두 에이전트는 직접적으로 서로의 코드를 호출하지 않습니다. 대신 **Blackboard(판)**이라는 공유 상태 파일을 통해 비동기적으로 통신합니다.

1. Hermes가 작업을 완료하면 Blackboard에 결과 파일 경로를 기록합니다.
2. OpenClaw는 주기적으로 Blackboard를 스캔하여 변경 사항을 감지하고, 백업 및 동기화를 실행합니다.

이러한 방식은 Hermes의 작업을 방해하지 않으면서도 시스템 관리를 지속할 수 있게 해줍니다.

---

## 결과 및 한계
두 에이전트를 분리한 후, 시스템 관리로 인한 사용자 작업 인터럽트가 대폭 줄었습니다. 다만, "두 에이전트가 서로 다른 사실을 기억하고 있어 충돌한다"는 **상태 불일치(State Drift)** 문제가 발생했습니다. 이는 Blackboard의 원자성(Atomic Write) 강화로 해결되었습니다.
