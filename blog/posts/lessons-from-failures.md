# 실패 패턴에서 배운 교훈

> 태그: #lessons-learned #debugging
> 읽는 시간: ~5분

## TL;DR
완벽한 시스템은 없습니다. Hermes도 수많은 오류와 실패를 겪었습니다. 이 포스트에서는 반복적으로 발생했던 "대형 참사"들과 그 근본 원인(Root Cause), 그리고 이를 시스템적으로 막기 위해 도입한 안전 장치를 공유합니다.

---

## 1. 중복 폴더 생성 문제 (JOB-907, JOB-1143)
- **현상**: 에이전트가 작업 폴더를 생성할 때, `~/.hermes/workspace/jobs/` 대신 임의의 경로를 만들었습니다. 
- **문제**: 워크플로우 관리 스크립트가 첫 번째 폴더만 인식해 결과가 분산되었고, 데이터 동기화가 깨졌습니다.
- **해결**: **JOB 등록 스크립트(`create-job.sh`) 강제화**. 에이전트는 직접 `mkdir`을 사용할 수 없으며, 오직 승인된 스크립트를 통해 폴더를 생성합니다.

## 2. 심링크(Symlink) 폭주 (JOB-1626)
- **현상**: 두 에이전트(Hermes와 OpenClaw)의 파일을 동기화하기 위해 심링크를 광범위하게 사용했습니다.
- **문제**: LLM의 파일 탐색 효율이 급격히 저하되었고, "파일은 있는데 왜 못 읽어?"라는 오류가 빈번했습니다.
- **해결**: **물리적 격리 원칙 도입**. 심링크를 금지하고, 대신 상태 파일과 이벤트 기반으로 비동기 동기화.

## 3. 컨텍스트 손실 (Context Collapse)
- **현상**: 장시간 세션 중반에 에이전트가 "설계서 내용을 잊어버리고" 직접 코드를 수정하려 했습니다.
- **문제**: 컨텍스트 윈도우 한계에 도달하거나 컴팩션(Compaction) 과정에서 핵심 가이드라인이 누락되었습니다.
- **해결**: **SSOT(Single Source of Truth) 설계서**. 에이전트가 중간에 잊어버려도, `design.md` 파일을 매번 다시 읽게끔 강제했습니다.

## 4. Gateway Hook 반환 형식 오류 (JOB-1233)
- **현상**: Gateway 스크립트가 "NO_REPLY"라는 텍스트를 반환하여 메시징 시스템이 뻑뻑해졌습니다.
- **문제**: 시스템은 텍스트가 아닌 `{"action": "skip"}`과 같은 딕셔너리 형식을 기대하고 있었습니다.
- **해결**: **Type Enforcement**. 스크립트 반환 형식을 코드 레벨에서 강제 검증하여 텍스트 반환 시 즉시 예외 처리.

---

## 교훈: 반복되는 실수를 시스템으로 막아라
어떤 문제든 "에이전트가 더 똑똑해지면 해결된다"는 생각은 버렸습니다. 
1. **규칙을 텍스트로 쓰지 않는다** (스크립트로 강제한다).
2. **수동 검증을 신뢰하지 않는다** (자동 검증을 실행한다).
3. **실패 시 재시도하지 않는다** (근본 원인을 분석하고 구조를 고친다).
