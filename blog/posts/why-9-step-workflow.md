# 왜 9단계 상태머신인가?

> 태그: #workflow #architecture
> 읽는 시간: ~10분

---

## TL;DR

AI 에이전트에게 "무조건 빨리 해라"는 것은 "무조건 빨리 망해라"와 다름없습니다. Hermes의 9단계 상태머신은 **에이전트의 무분별한 실행을 차단**하고, 사용자의 통제권을 확보하며, 시스템적인 품질 관리를 위해 설계되었습니다.

```
┌─────────────────────────────────────────────────────┐
│              9단계 상태머신                          │
├─────────────────────────────────────────────────────┤
│                                                     │
│  1. Request  →  2. Investigation  →  3. Design     │
│  4. Review   →  5. Approval     →  6. Execution    │
│  7. Test     →  8. Execution Review → 9. Done      │
│                                                     │
└─────────────────────────────────────────────────────┘
```

---

## 배경: "실행의 함정"

### 초기 버전의 문제

2025년 초, Hermes 에이전트는 사용자의 요청을 받으면 즉시 코드를 실행하거나 파일을 수정했습니다. 이로 인해 여러 가지 치명적인 문제가 발생했습니다.

### 3가지 치명적 문제

**1. 컨텍스트 미스 (Context Miss)**
- 에이전트가 파일의 전체 구조를 파악하지 않고 일부만 수정하여 시스템이 깨지는 현상
- 예시: `config.yaml`의 1개 필드만 수정하려다, 전체 YAML 구조 파손
- 결과: 시스템 설정 파일 손상, 수동 복구 필요

**2. 롤백 불가 (Irreversible)**
- 파일이 덮어씌워진 후, "이게 아닌데"라고 해도 원상 복귀가 불가능한 상황
- 예시: Python 스크립트 전체 재작성 → 구버전 백업 부재
- 결과: 2시간 작업 손실

**3. 에이전트 폭주 (Agent Rampage)**
- 복잡한 지시(예: "아키텍처를 재설계해줘")를 받아들이다가 수십 개의 파일을 동시에 망가뜨리는 사고
- 예시: 2025-09-15, `workflow.md` 15개 파일 동시 수정 시도 → 전체 시스템 고장
- 결과: 6시간 복구 작업

**"더 똑똑한 모델을 넣으려면?"이라는 해결책은 근본 원인을 치료하지 못했습니다. 문제는 지능이 아닌 프로세스의 부재였습니다.**

## 설계 결정: 9단계 파이프라인

Hermes는 모든 복잡한 작업을 강제적인 9단계로 쪼개어 처리합니다.

### 상태 전이 다이어그램

```
┌──────────┐    ┌──────────┐    ┌──────────┐
│ Request  │    │Investigation│   │  Design  │
│ (수신)   │───→│ (조사)    │───→│ (설계)   │
└──────────┘    └──────────┘    └──────────┘
                                              │
┌──────────┐    ┌──────────┐    ┌──────────┐ │
│ Done     │    │Exec Review│    │  Test    │◄┘
│ (완료)   │←───│(검증)    │←───│(테스트)  │
└──────────┘    └──────────┘    └──────────┘
     ↑                   │              │
     │                   │              │
┌──────────┐    ┌──────────┐    ┌──────────┐
│Approval  │    │Execution │    │  Review  │
│ (승인)   │───→│ (실행)   │───→│(리뷰)    │
└──────────┘    └──────────┘    └──────────┘
```

### 각 단계 상세 설명

| 단계 | 이름 | 설명 | 산출물 |
|------|------|------|--------|
| 1 | Request | 사용자 요청 수신 | `request.md` |
| 2 | Investigation | 현황 분석, 문제점 식별 | `investigation.md` |
| 3 | Design | 상세 설계서 작성 | `design.md` |
| 4 | Review | 설계서 검증 (Self-Review) | `review.md` |
| 5 | Approval | 사용자 승인 요청 | - |
| 6 | Execution | 실제 코드/파일 수정 | `execution.log` |
| 7 | Test | 테스트 실행, 결과 검증 | `test-results.json` |
| 8 | Execution Review | 실행 결과 최종 검증 | `execution-review.md` |
| 9 | Done | 작업 완료 보고 | `result.md` |

### 핵심 분리: Investigation → Design → Execution

**1. Investigation (조사)**: "무엇을 할지"를 파악하기 전에, "지금 시스템이 어떤 상태인지"를 먼저 봅니다.

```python
# investigation.md 예시 구조
# JOB-1001 조사 리포트

## 1. 문제 정의
## 2. 현황 분석
## 3. 권장 해결 방안
## 4. 결론
```

**2. Design (설계)**: 코드를 작성하기 전에 `design.md`라는 상세 설계서를 먼저 작성합니다.

```python
# design.md 예시 구조
# JOB-1001 설계서

## 개선 항목별 상세 설계
### §1: 기능 정의
### §2: API 변경
### §3: 테스트 계획
## 실행 순서
```

**3. Execution (실행)**: 설계서 기반으로 실제 코드/파일 수정

### 왜 하필 9단계인가?

9단계는 **최소한의 안전 장치를 모두 갖춘 가장 작은 숫자**입니다.

- `Review` 단계가 두 번 (설계 후, 실행 후) 들어간 이유: **자신감은 검증되지 않습니다.** 설계가 완벽해도 실행은 실패할 수 있기 때문입니다.
- `Test` 단계가 별도로 존재하는 이유: 코드의 정확성은 테스트로만 검증 가능합니다.

## 다른 대안과의 비교

| 접근법 | 문제점 | Hermes 해결책 |
|--------|--------|---------------|
| **Prompt Engineering** | 모델의 상태에 따라 결과 불안정 | 상태머신으로 프로세스 자체를 안정화 |
| **Unstructured Chat** | 롤백 불가능, 추적 어려움 | JOB 폴더 및 상태 파일 저장 |
| **Single-Step Execution** | 컨텍스트 누락으로 인한 시스템 고장 | Investigation → Design 분리 |
| **Manual Review** | 인간 개입 필요, 느림 | 자동화된 9단계 상태머신 |

## 실제 운영 사례

### 성공 사례: 복잡한 아키텍처 변경 (JOB-1626)

**작업**: 5-Tier 물리 계층화
**단계별 진행**:
1. Request: "심링크 폭주 문제 해결 필요"
2. Investigation: 심링크 사용 현황 분석 → 66개 스크립트 경로 수정 필요
3. Design: 5-Tier 구조 설계, 폴더 분리 계획
4. Review: 설계서 검증 → 100% 준수 확인
5. Approval: 사용자 승인 → "진행해"
6. Execution: 실제 폴더 분리, 스크립트 수정
7. Test: 66개 스크립트 실행 테스트 → 66/66 PASS
8. Execution Review: 최종 검증 → 100% 준수
9. Done: 작업 완료 보고

**결과**: 심링크 폭주 문제 해결, 0회 재작업

### 실패 사례: 단순 요청에 과도한 프로세스

**문제**:
- "약간 수정해 줘"라는 간단한 요청에도 9단계 진행
- 결과: 15분 소요 → 사용자 불만

**해결**:
- `[TASK]` (단순 작업)과 `[JOB-XXXX]` (공식 프로젝트) 분리
- 가벼운 요청은 워크플로우 우회 가능
- 9단계는 복잡한 작업에만 적용
