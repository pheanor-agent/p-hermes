# 이벤트 기반 도메인 통신

> 태그: #architecture #workflow
> 읽는 시간: ~4분

## TL;DR
시스템이 복잡해지면, 각 컴포넌트가 서로를 직접 호출하게 됩니다. 이는 치명적인 결합을 의미합니다. Hermes는 **"도메인 간 직접 호출을 금지"**하고, 대신 상태 파일과 이벤트를 통한 비동기 통신을 채택했습니다.

---

## 배경: "동기 호출의 지옥"
초기 시스템에서 지식 업데이트 스크립트가 실행되면, 이 스크립트가 직접 백업 스크립트를 호출했습니다. 백업 스크립트가 끝나면 다시 세션 정리 스크립트를 호출했습니다.

문제는 무엇일까요?
- **블로킹(Block)**: A가 B를 호출하면, A는 B가 끝날 때까지 멈춰 서 있어야 합니다.
- **연쇄 반응**: B에서 오류가 발생하면 A도 함께 죽습니다.
- **동시성 붕괴**: 두 에이전트가 동시에 하나의 스크립트를 호출하면 데이터가 망가집니다.

---

## 설계 결정: 상태 파일 기반 이벤트 통신

Hermes는 "함수 호출"을 "파일 변경 이벤트"로 대체했습니다.

### 1. 직접 스크립트 실행 금지
도메인 A의 스크립트가 도메인 B의 스크립트를 `bash b.sh`로 호출하는 것을 원칙적으로 금지합니다. (단, Gateway 스크립트는 예외)

### 2. 상태 파일 작성 (Atomic Write)
A가 작업을 마치면, 공유 폴더에 `event.json`이나 `.workflow-state` 파일의 상태를 갱신합니다.
```json
{
  "jobId": "JOB-1001",
  "status": "done",
  "artifacts": ["path/to/result.md"]
}
```

### 3. 비동기 감지 및 처리
다른 도메인(예: OpenClaw 관리자 에이전트)은 주기적으로 해당 폴더를 스캔합니다. 파일이 변경되면 변경된 내용을 읽고 자신의 작업을 수행합니다.

---

## 다른 대안과의 비교

| 대안 | 문제점 | 이벤트 기반 통신 |
|---|---|---|
| **함수/스크립트 직접 호출** | 결합도가 높고 오류가 연쇄 전파됨 | 도메인 간 완전한 비동기 격리 |
| **Message Queue (RabbitMQ 등)** | 운영 및 유지보수가 너무 복잡함 | 파일 시스템 자체를 큐로 활용 |
| **DB Trigger** | 데이터베이스 스키마 변경이 필요함 | 별도의 인프라 설정 불필요 |

---

## 결과 및 한계
이벤트 기반 통신은 시스템의 결합도를 극단적으로 낮췄습니다. 하지만 "파일 변경이 감지되기도 전에 다른 에이전트가 같은 파일을 덮어쓴다"는 경합 조건(Race Condition)이 발생할 수 있습니다. 이 문제는 **`flock` 기반의 Atomic Write**로 해결되었습니다.
