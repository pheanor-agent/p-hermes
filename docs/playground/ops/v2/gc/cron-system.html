🔴 WHY — 왜 자동화인가?

rule: "3열 비교 카드 (문제 명확히)" · audience: developer · tone: educational

🧑‍💻

수동 실행의 한계

에이전트 세션이 종료되면 모든 작업 정지. 매일 아침 뉴스 요약 같은 루틴을 사용자가 매번 요청해야 함.

  • 사용자 개입 필수
  • 휴일/야간 실행 불가
⏰

시간 민감 작업

정해진 시간에 정확히 실행되어야 하는 작업 — 주간 점검, 데이터 백업, 로그 회전은 지연이 허용 안 됨.

  • 기억 의존 → 누락 필연
  • 일정 충돌 발생
👁️

감시 필요성

시스템 상태, API 응답 시간, 디스크 사용량 — 지속적 감시 없이는 문제를 조기에 발견할 수 없음.

  • 실패해도 아무도 모름
  • 재시도도 알림도 없음
💡 WHAT · section_flow[2]

💡 WHAT — 핵심 구성 요소

rule: "3열 card-grid (기술 개념)" · tone: educational · code level: full

🗓️
cron-job 스케줄러

registry.yaml이 SSOT. cron 표현식 또는 ISO 8601로 주기/단회 실행 정의.

schedule: "0 8 * * *" # 매일 8시 schedule: "2026-07-01T14:30:00" # 단회
📡
이벤트 버스 (event.sh)

모든 실행 이력을 JSONL로 기록. started → completed → delivered 전 과정 추적 가능.

event.sh publish --job morning-news --event started event.sh history --limit 10 event.sh history --job JOB-1234
🛡️
Watchdog (no_agent)

LLM 오버헤드 제로. 스크립트 stdout이 비어 있으면 silent, 존재하면 알림 전송.

no_agent: true # LLM bypass script: "scripts/check.sh" # stdout 비어있음 → silent
🔧 HOW · section_flow[3]

🔧 HOW — 실제 사용법

rule: "단계별 플로우 + CLI 예제" · tone: educational · code level: full

1
JOB 등록 — action=create

LLM Driven Job 또는 Script Driven Job 선택. 프롬프트로 복잡한 추론, 스크립트로 경량 실행.

# LLM 구동 — 매일 9시 작업 리포트 hermes cron create \ --name "daily-briefing" \ --prompt "오늘 주요 작업 확인" \ --schedule "0 9 * * *"
2
스크립트 연결 — script=

Watchdog 패턴: stdout 출력이 있으면 알림 전송, 없으면 silent. mutex_acquire로 중복 방지.

# Script Driven — 5분마다 Wiki 동기화 hermes cron create \ --name "wiki-sync" \ --script "scripts/wiki-sync.sh" \ --schedule "every 5m" \ --no_agent
3
배포 채널 지정 — deliver=

결과를 특정 채널로 전송. origin은 작업이 시작된 대화 창, all은 모든 연결 채널.

# 배포 채널 예시 deliver: "origin" # 현재 대화 창 deliver: "origin,all" # 현재 + 모든 채널 deliver: "discord:123456" # 특정 채널

⚖️ TRADE-OFFS · section_flow[4]

⚖️ Trade-offs — 장점과 한계

rule: "2열 pros/cons 비교" · tone: balanced

✅ 장점

  • 완전 자동화 — 사용자 개입 제로. 설정만 하면 24/7 실행.
  • 경량 실행 — no_agent 모드에서 LLM 오버헤드 없음. 스크립트만 실행.
  • 병렬 처리 — 독립적인 Job이 동시에 실행 가능. 서로 블로킹 없음.
  • silent-on-success — 정상 시 알림 없음. 실패/경고 시만 전달.

⚠️ 한계

  • cron 의존성 — system crontab 또는 OpenClaw cron에 의존. 둘 중 하나가 죽으면 전체 중단.
  • 모니터링 필요 — Job이 실패해도 자동으로 알려주지 않으면 방치됨. event.sh history 주기적 확인 필요.
  • 자동 재시도 없음 — 한 번 실패한 Job은 다음 스케줄까지 재실행 안 됨. 수동 개입 필요.
  • LLM 비결정성 — 같은 프롬프트도 매번 다른 결과. 스크립트 기반보다 예측 불가.
📝 SUMMARY · section_flow[5]

📝 3줄 요약

1️⃣ 24시간 자동화
cron-job 스케줄러가 LLM 구동 Job과 Script 구동 Job을 주기적으로 실행. 사용자는 아침에 결과만 확인.
2️⃣ event.sh 단일 진입점
모든 실행 이력은 event.sh를 통해 JSONL로 기록. started → completed → delivered 전 과정 추적 가능. 디버깅과 Audit의 SSOT.
3️⃣ Watchdog 패턴 (no_agent)
LLM 오버헤드 제로. stdout 출력이 비어 있으면 silent, 존재하면 알림. mutex_acquire로 중복 실행 차단.
