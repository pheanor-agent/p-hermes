Pre-Direction Engine

5-Layer Validator
L1~L5 계층 검증 게이트.

🔧 HOW — 3단계 워크플로우

2-Phase (사전 방향 + 사후 검증), D1~D5 전 도메인, 계층적 검증

🟦 Pre-Phase → 🟩 Guide-Phase → 🟧 Post-Phase
1
방향성 생성

pre_direction → direction-guide.json 생성.
목적·청중·톤·포맷·최대 단어수 등 사전 정의

2
작성 가이드

strategy + audience + template 조합.
direction-guide 기반으로 콘텐츠 생산 실행

3
품질 검증

validator L1→L5 순차 통과.
실패 계층에서 재수정, 전체 통과 시 최종 승인

🟦 Pre-Phase (사전 방향성)

pre_direction → direction-guide.json. 작성 전에 모든 방향성 정의.
quality_by_design 원칙 — 사전에 잘못된 방향 차단

🟧 Post-Phase (사후 검증)

L1→L5 계층 검증. L3 Antislop이 핵심 계층. L5 Judge가 최종 심사.
quality_by_inspection 원칙 — 사후에 품질 보증

section_flow[4] · TRADE-OFFS

⚖️ Trade-offs

2열 pros/cons 비교 — 체계적 품질 vs 초기 설정 비용

✅ 장점

  • 체계적 품질 관리 — 사전 방향 + 사후 검증 2-Phase로 일관된 품질 보증
  • 재현 가능 — direction-guide.json이 SSOT, 동일 설정으로 동일 품질 재현
  • 도메인별 맞춤 — D1~D5 각 도메인에 최적화된 기준과 검증 적용

⚠️ 한계

  • 초기 설정 복잡 — direction-guide 스키마, L1~L5 규칙, 도메인 매트릭스 초기 정의 부담
  • 템플릿 의존성 — direction-guide가 없으면 생산 불가. 가이드 작성 자체가 추가 작업
  • 엔진 유지보수 — 5계층 Validator + Domain Wrapper + Pre-Direction Engine 지속 업데이트 필요
section_flow[5] · SUMMARY

📝 3줄 요약

1️⃣ 사전 방향 + 사후 검증 2-Phase

Pre-Direction Engine으로 작성 전 방향 설정 → 5-Layer Validator로 사후 검증.
quality_by_design + quality_by_inspection 통합

2️⃣ D1~D5 전 도메인 지원

기술 문서(D1)부터 비즈니스(D5)까지 5개 도메인 각각 최적화된 가이드라인과 검증 기준

3️⃣ 계층적 검증 (L1→L5)

구문→오류→Antislop→도메인→Judge 순차 검증.
각 계층 독립 동작, 실패 시 재수정 자동화
