GOLDEN CIRCLE · STEP 1
WHY · 문제
왜 명세서가 먼저인가?
명세서 없이 코딩하면 세 가지 문제가 반복됩니다.
추측, 격차, 그리고 망각 — 이들이 소프트웨어 품질을 서서히 갉아먹습니다.
PROBLEM 01

추측 기반 코딩

명세 없이 시작하면 요구사항이 불일치합니다. 개발자의 '추측'이 실제 요구를 대체하고, 결과물은 항상 빗나갑니다.

명세 없음 → 요구사항 불일치
PROBLEM 02

문서와 코드 격차

구현 후 문서는 항상 구식이 됩니다. 코드는 진화하지만 문서는 업데이트되지 않아, 유지보수는 점점 어려워집니다.

구현 완료 → 문서 누락
PROBLEM 03

재현 불가능

왜 그렇게 구현했는지 기록이 없습니다. 결정의 맥락이 사라지면 버그 수정도, 기능 추가도 '재발견'이 됩니다.

왜 그랬지? — 기록 없음
GOLDEN CIRCLE · STEP 2
WHY · 비용과 영향
문제가 초래하는 비용
세 가지 문제는 단순한 불편이 아니라, 조직과 제품에 실질적인 손실을 일깁니다.
🔄

잦은 재작업

요구사항 불일치로 인한 재작업이 개발 시간의 40%를 차지합니다. 명세가 없으면 '맞추기'보다 '다시 만들기'가 기본이 됩니다.

40% 시간 낭비
🧩

유지보수 부채

문서와 코드의 격차가 쌓이면 유지보수 비용이 기하급수적으로 증가합니다. 신규 개발자의 온보딩 시간도 2배 이상 늘어납니다.

2배 온보딩 비용
🔍

맥락 손실

6개월 후에는 '왜 이렇게 했는지' 아무도 기억하지 못합니다. 기록 없는 결정은 기술 부채로 전환되어 복구 불가능해집니다.

기술 부채 누적

Spec = SSOT
Single Source of Truth. 모든 변경은 Spec을 경유합니다.
Spec이 먼저, 코드는 그 다음 — 이 순서가 품질을 결정합니다.
S

Spec = SSOT

모든 변경은 반드시 Spec을 통과해야 합니다. 코드는 Spec의 구현체일 뿐, Spec이 진정한 설계 산출물입니다.

9

9단계 연동

설계 단계에서 Spec design.md를 자동 생성합니다. 9단계 상태머신의 설계 게이트가 Spec의 완전성을 검증합니다.

✦

Spec Matrix

의존성 그래프와 충돌 감지를 자동화합니다. Spec 간의 관계를 명시적으로 관리하여 숨은 의존성을 사전에 발견합니다.

GOLDEN CIRCLE · STEP 4
WHAT · 시스템
Spec 중심 개발의 세 가지 축
SSOT, 9단계, Matrix — 세 가지 메커니즘이 Spec 중심 개발을 실현합니다.
📋

변경은 Spec 경유

코드를 직접 수정하지 않습니다. 먼저 Spec에 변경을 문서화하고, 승인된 후에만 코드를 변경합니다.

  • 변경 의도와 범위를 Spec에 기록
  • 리뷰를 통해 논리적 타당성 검증
  • 승인 후 Spec에 따라 코드 구현
🔗

9단계 + design.md

설계 단계에서 자동으로 Spec design.md가 생성됩니다. 이후 모든 게이트(gate)가 Spec의 완전성을 확인합니다.

  • 설계 단계 진입 시 Spec design.md 자동 생성
  • 게이트 검증이 Spec 누락 탐지
  • Spec 불완전 시 다음 단계 차단
🧿

Spec Matrix

다중 Spec 간의 의존성을 그래프로 관리합니다. 충돌하는 Spec을 자동 감지하고, 변경 영향 범위를 시각화합니다.

  • 의존성 그래프 자동 구축
  • Spec 간 충돌 감지 및 경고
  • 변경 영향 범위 시각화

HOW · 워크플로우
Spec → Review → Code
세 단계로 구성된 간결한 워크플로우. Spec과 코드의 1:1 대응이 핵심입니다.
✍ Spec

Spec 작성

변경할 내용을 먼저 문서화합니다. 요구사항, 설계 결정, 예상 영향 범위를 명확히 기록합니다.

  • 변경 의도와 배경 서술
  • 영향 받는 모듈 식별
  • 예상 출력/동작 정의
✓ Review

논리적 리뷰

Spec의 논리적 타당성을 검증합니다. 설계 결정이 올바른지, 누락된 케이스는 없는지 검토합니다.

  • 요구사항 일치 여부 확인
  • 에지 케이스 누락 검사
  • 기존 Spec과의 충돌 확인
▸ Code

코드 생성 (1:1)

Spec과 1:1로 대응하는 코드를 생성합니다. Spec의 각 항목이 코드의 특정 부분에 직접 연결됩니다.

  • Spec 항목별 코드 모듈 매핑
  • Spec 변경 시 코드 재생성
  • 트레이서빌리티 자동 유지
GOLDEN CIRCLE · STEP 6
BALANCE · 트레이드오프
장점, 한계, 그리고 결론
모든 방법론은 트레이드오프를 수반합니다. 투명하게 공개하고 균형을 유지합니다.

✓ Pros

끝까지 요구사항 추적 — 변경의 시작과 끝을 Spec으로 추적합니다. "왜 이렇게 했는지"가 항상 명확합니다.
일치 코드+문서 일치 — 코드와 문서가 항상 동기화됩니다. Spec이 SSOT이므로, 문서가 구식이 될 수 없습니다.
재현 재현 가능 — 모든 결정의 맥락이 Spec에 기록됩니다. 6개월 후에도 결정 이유를 정확히 알 수 있습니다.

✗ Cons

비용 초기 문서화 비용 — 코딩 전에 Spec을 작성해야 하므로 초기 진입 장벽이 있습니다. 단순 버그 수정에도 Spec이 필요합니다.
재검토 Spec 변경 시 전면 재검토 — Spec이 바뀌면 전체 설계를 재검증해야 합니다. 빠른 실험/프로토타이핑에 부적합합니다.
과잉 과잉 문서화 위험 — 모든 변경을 Spec으로 관리하면 문서가 코드보다 많아질 수 있습니다. 적절한 선 긋기가 필요합니다.
