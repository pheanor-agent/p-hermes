WHY · 문제
왜 필요한가?
AI 에이전트 작업은 되돌릴 수 없는 위험을 수반합니다.
명확한 관문(gate) 없이는 작은 실수가 큰 재앙으로 이어집니다.
PROBLEM 01

Context Miss

들여쓰기 실수가 전체 시스템 부팅 실패로 이어짐. 에이전트는 컨텍스트 윈도우 밖의 의존성을 감지하지 못함.

들여쓰기 실수 → 부팅 실패
PROBLEM 02

Irreversible

되돌릴 수 없는 변경. 수백 줄 재작성 과정에서 예외 처리나 에지 케이스가 증발. 이전 상태로의 복원이 불가능.

수백 줄 재작성 → 예외 증발
PROBLEM 03

Agent Rampage

15개 파일을 무차별적으로 수정하고 롤백에 6시간 소요. 에이전트가 통제 불능 상태가 되어 전체 코드베이스를 위험에 빠뜨림.

15개 파일 수정 → 6시간 복구
WHAT · 해결책
무엇을 만들었는가?
상태 전이가 검증된 9단계 상태머신 — 각 단계마다 산출물 검증과 게이트를 통과해야만 다음으로 진행.
9

9단계 상태머신

초기화부터 마무리까지 9개의 명확한 상태. 각 상태는 정의된 입·출력을 가지며 인접 상태로만 전이 가능.

✓

상태 전이 검증

workflow-gate.sh가 각 전이의 유효성을 검증. 허용되지 않은 전이는 차단되고 명확한 오류 메시지를 반환.

📦

산출물 강제

각 상태는 반드시 산출물(deliverable)을 생성해야 함. 산출물이 없으면 다음 상태로 전이 불가.

init → analyze → gate → design → gate → develop → gate → review → finish
HOW · 사용법
어떻게 사용하는가?
세 가지 명령어로 전체 워크플로우를 제어합니다. 복잡할 것 없이 — 상태 확인, 전이, 검증.
--status

상태 확인

현재 상태와 허용된 전이 목록을 표시합니다. 항상 어디에 있는지 알 수 있습니다.

  • 현재 상태 하이라이트
  • 가능한 다음 전이 표시
  • 산출물 상태 요약
transition

전이 실행

명시적인 전이 명령으로만 다음 상태로 이동합니다. 자동 전이는 없습니다.

  • 전이 전 게이트 검증
  • 산출물 존재 확인
  • 실패 시 명확한 사유 반환
validate

산출물 검증

workflow-gate.sh가 산출물의 완전성과 품질을 자동 검증합니다.

  • 파일 존재 및 형식 확인
  • 필수 필드 누락 검사
  • 상태별 규칙 위반 탐지
BALANCE · 트레이드오프
장점과 한계
모든 설계 결정은 트레이드오프를 수반합니다. 투명하게 공개합니다.

✓ Pros

0건 도입 후 사고 0건 — 게이트 검증으로 모든 상태 전이가 안전함이 보장됨
2.7→0.4 재작업 횟수 2.7회에서 0.4회로 감소 — 검증된 산출물로 후행 작업 안정화
47%→7% 롤백 비율 47%에서 7%로 감소 — 되돌릴 수 없는 변경의 사전 차단

✗ Cons

2주 학습 곡선 — 새 상태머신 적응에 약 2주 소요. 초기 생산성 저하 불가피
오버헤드 단순 작업(파일 1~2개 수정)에도 9단계를 거쳐야 함. 가벼운 작업에는 과도한 절차
강제성 9단계 준수는 선택이 아닌 강제. 일부 팀원에게는 유연성 부족으로 느껴질 수 있음
SUMMARY · 요약
결론
모든 사고의 공통 원인은 관문(gate)의 부재였습니다. 9단계 상태머신이 그 빈틈을 메웁니다.
사고 원인 = 관문 부재
