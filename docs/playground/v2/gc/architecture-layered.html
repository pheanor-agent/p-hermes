WHY — Step 1

문제: 모듈 간 의존성

수정 시 연쇄 파괴

🔗 순환 의존성
Core와 Interface 모듈이 서로 참조
A 수정 → B 오류 → A 재수정 (무한 루프)
💥 연쇄 변경
한 모듈 수정 시 관련 모듈 3~5개 추가 수정 필요
배포 일정 지연, 회귀 버그 증가

WHY — Step 2

영향: 계층 분리 부재

Core + Interface 혼재

🧩 Core + Interface 혼재
핵심 로직(Core)과 외부 연결(Interface)이
동일 모듈에 공존 → 수정 범위 폭발
🧱 확장성 부족
새 Interface(WebSocket, IPC 등) 추가 시
Core 전체 수정 필요 → 대규모 리팩토링
🔄 심링크(Symlink) 오염
$HERMES_ROOT 없이 절대 경로 심링크 사용
→ 환경 이전 시 참조 붕괴

HOW — Step 3

접근: 계층 식별 & 의존성 검증

계층 식별 & 단방향 의존성

🔍 계층 식별 (Layer Identification)
모든 모듈을 관심사(concern)별로 분류
Core → Runtime → Interfaces → Infra → Release
⬇ 단방향 의존성 규칙
✅ 상위 계층이 하위 계층만 참조
❌ 하위 계층이 상위 계층 참조 금지
위반 시 컴파일/빌드 실패
🔐 $HERMES_ROOT 추상화
심링크 금지, 환경 변수 기반 경로 참조
환경 독립적 이식성 확보

HOW — Step 4

실행: 인터페이스 분리 & 검증

인터페이스 분리 패턴

🔌 계층 간 통신 (Interface Segregation)

공개 API + 내부 구현 분리
각 계층은 공개 인터페이스(Public API)만 노출
내부 구현 변경이 외부에 영향 없음
의존성 검증 자동화
CI 단계에서 dep-validator 실행
역참조 발견 시 빌드 실패 → 조기 차단

WHAT — Step 5

구조: 5-Tier 아키텍처

5-Tier 계층 구조

🚀 Release 배포 · CI/CD · 버전 관리
⬆ 의존
🗄️ Infrastructure DB · 네트워크 · 파일 시스템
⬆ 의존
🌐 Interfaces API · WebSocket · IPC · CLI
⬆ 의존
⚙️ Runtime 실행 엔진 · 스케줄러 · 워커
⬆ 의존
🧠 Core 핵심 로직 · 도메인 모델
📐 설계 원칙
✅ 단방향 의존성
하위 계층만 참조, 역참조 금지

✅ 관심사 분리 (SoC)
각 계층은 하나의 책임

✅ $HERMES_ROOT 추상화
심링크 금지, 환경 변수 기반

데이터 흐름: End-to-End

        sequenceDiagram
          participant C as Client
          participant I as Interfaces
          participant R as Runtime
          participant Co as Core
          participant Inf as Infrastructure

          C->>I: HTTP/WebSocket 요청
          I->>R: 명령 디스패치
          R->>Co: 도메인 로직 실행
          Co->>Inf: 데이터 조회/저장
          Inf-->>Co: 결과 반환
          Co-->>R: 처리 결과
          R-->>I: 응답 변환
          I-->>C: 응답 전송

모든 통신은 계층 경계를 따라 단방향으로 흐름

TRADE-OFFS — Step 6

장단점 & 요약

Trade-offs

✅ 장점 (Pros)

항목효과
유지보수 용이계층별 수정 → 타 계층 영향 Zero
독립 배포계층별 단위 배포 가능
계층별 테스트Mock/Stub으로 계층 단위 검증
확장성새 Interface 추가 시 Core 불변

⚠️ 단점 (Cons)

항목영향
초기 설계 비용계층 분할 + 인터페이스 정의 시간
계층 통과 필요기능 추가 시 N개 계층 변경
인터페이스 오버헤드추상화 계층에 의한 성능 저하
러닝 커브팀 전체 계층 규칙 숙지 필요
