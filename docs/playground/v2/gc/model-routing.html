GOLDEN CIRCLE · STEP 1
WHY · 문제
왜 Model Routing이 필요한가?
하나의 모델로 모든 작업을 최적으로 처리할 수 없습니다.
작업 유형별 특화 모델이 필요하지만, 이를 수동으로 관리하면 복잡도가 폭발합니다.
PROBLEM 01

모델별 특화

모든 작업에 하나의 모델로 최적의 결과를 내는 것은 불가능합니다. 코딩, 추론, 창작 등 각 작업에 특화된 모델이 필요하지만 수동 전환은 비효율적입니다.

하나의 모델로 모든 작업 → 최적 불가
PROBLEM 02

비용 효율

고성능 모델은 비용이 높습니다. 단순한 질문에도 고성능 모델을 사용하면 불필요한 비용이 발생합니다. 작업 난이도에 맞는 모델 선택이 필요합니다.

단순 작업에 고성능 모델 → 비용 낭비
PROBLEM 03

가용성 관리

단일 모델에 의존하면 장애 발생 시 전체 시스템이 마비됩니다. 모델 제공자 장애, Rate Limit, 네트워크 문제 등에 대응할 Fallback 체계가 필수입니다.

단일 모델 의존 → 장애 시 전체 마비
GOLDEN CIRCLE · STEP 2
WHY · 영향
세 가지 문제가 초래하는 손실
모델 선택의 비효율성은 단순한 불편을 넘어, 비용 증가와 서비스 불안정이라는
실질적인 손실로 이어집니다. Model Routing이 이 모든 문제를 해결합니다.
💰

비용 증가

고성능 모델을 모든 작업에 사용하면 API 비용이 급증합니다. 코딩 작업에는 Claude, 추론에는 DeepSeek, 창작에는 Qwen 등 각 작업에 최적화된 모델로 비용을 절감할 수 있습니다.

최대 70% 비용 절감 가능
⏱️

서비스 불안정

모델 제공자의 장애나 Rate Limit에 대응할 Fallback이 없으면 서비스 중단이 발생합니다. 빠른 모델(fast)과 정확한 모델(reasoning)을 구분하여 응답 시간과 품질을 동시에 확보해야 합니다.

GOLDEN CIRCLE · STEP 3
WHAT · 해결책
Role 기반 라우팅 + 2계층 설정
Model Routing은 작업 유형(Role)에 따라 최적의 모델을 자동 선택합니다.
2계층 설정(SSOT + Override)으로 유연성과 일관성을 동시에 확보합니다.
R

Role 기반 라우팅

작업 유형별로 Role을 정의합니다. default / coding / reasoning / creative / fast / vision — 6개 Role이 각각 최적의 모델에 매핑되어 자동 라우팅됩니다.

  • coding → Claude Sonnet
  • reasoning → DeepSeek R1
  • fast → GPT-4o Mini
2

2계층 설정

SSOT model-roles.yaml이 단일 진실 공급원으로 모든 Role-모델 매핑을 정의합니다. config.yaml에서 프로필별 오버라이드로 유연하게 커스터마이징합니다.

  • SSOT: model-roles.yaml (팀 공통)
  • Override: config.yaml (개인 설정)
  • 계층 구조로 충돌 해소

GOLDEN CIRCLE · STEP 5
HOW · 적용 방법
3단계 설정 워크플로우
복잡해 보이지만 실제 설정은 단 3단계입니다.
역할 정의 → 모델 매핑 → 장애 대응, 이 순서대로만 따라가면 됩니다.
STEP 01

역할 정의

JOB 유형별로 필요한 역할(Role)을 정의합니다. 각 작업의 특성에 맞는 Role을 할당하여 최적의 결과를 보장합니다.

  • 작업 유형 분류 (coding / reasoning / creative 등)
  • 각 Role에 요구사항 명시
  • 우선순위 및 성능 기준 설정
STEP 02

모델 매핑

각 Role에 최적의 모델을 매핑합니다. model-roles.yaml에서 SSOT를 정의하고, 필요시 config.yaml로 오버라이드합니다.

  • Role별 1차/2차/3차 모델 지정
  • model-roles.yaml에 SSOT 기록
  • config.yaml로 개인 설정 오버라이드
STEP 03

장애 대응

Fallback 체인을 구성하여 장애에 대비합니다. 1차 모델 실패 시 자동으로 2차, 3차로 전환되어 서비스 연속성을 보장합니다.

  • Fallback 우선순위 설정
  • 장애 탐지 조건 정의
  • 복구 후 자동 재시도 정책

GOLDEN CIRCLE · STEP 6
BALANCE · 트레이드오프
장점과 한계
모든 설계 결정은 트레이드오프를 수반합니다. Model Routing의 장단점을 투명하게 공개합니다.

✓ Pros

최적 작업별 최적 성능 — 각 작업 유형에 가장 적합한 모델이 자동 선택되어 최고의 결과물 보장
절감 비용 효율 — 간단한 작업에는 경량 모델, 복잡한 작업에는 고성능 모델을 사용하여 API 비용 최적화
강건 장애 복원력 — Fallback 체인으로 단일 모델 장애가 전체 시스템에 영향을 주지 않음

✗ Cons

복잡 설정 복잡 — 2계층 설정(model-roles.yaml + config.yaml)의 이해와 초기 구성에 시간 필요
지연 라우팅 오버헤드 — Role 판별 및 Fallback 로직으로 인한 미세한 추가 지연 발생 (보통 수ms)
학습 모델 특성 파악 필요 — 각 모델의 강점과 약점을 이해해야 최적의 Role-모델 매핑이 가능
