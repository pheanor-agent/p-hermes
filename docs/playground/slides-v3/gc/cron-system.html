Cron System

24시간 자동화 — Hermes의 심장박동

cron-job 스케줄러 · event.sh 이벤트 버스 · Watchdog 패턴
WHY WHAT HOW
WHY · 1/3

🧑‍💻 수동 실행의 한계

AI 세션이 종료되면 모든 작업이 정지한다 — 매일 아침 뉴스 요약 같은 루틴을 매번 사용자가 직접 요청해야 한다.

생산성 저하 — 반복 작업을 사람이 직접 트리거해야 하므로 시간과 집중력이 낭비된다. 에이전트가 '자는' 시간 동안 아무 일도 일어나지 않는다.

WHY · 2/3

⏰ 시간 민감 작업의 위험

Healthcheck, 백업, 로그 회전 — 정해진 시간에 정확히 실행되어야 할 작업이 지연되거나 누락된다.

금요일 밤 백업이 누락되면 월요일 아침 복구 불가능. 기억에 의존한 스케줄 관리는 일정 충돌과 누락이 필연적이다.

WHY · 3/3

👁️ 감시 필요성 & 근본 원인

시스템 상태, 디스크 사용량, API 응답 — 지속적 감시 없이는 문제를 조기에 발견할 수 없다.

실패해도 아무도 모르고, 재시도도 알림도 없다. 문제를 발견했을 때는 이미 늦은 경우가 대부분.

🔑 근본 원인: 자동화 부재 — 반복·시간·감시 작업을 기계에 위임하지 않고 사람의 개입에 의존하는 구조 자체가 문제의 근원이다.

WHAT · 1/3

🗓️ cron-job 스케줄러

registry.yaml이 모든 Job 정의의 SSOT — cron 표현식 또는 ISO 8601로 주기/단회 실행을 정의한다.

🔄

주기 실행

"0 8 * * *" — 표준 cron 표현식으로 반복 작업 예약. LLM 구동 Job과 Script 구동 Job 모두 지원.

🎯

단회 실행

"2026-07-01T14:30:00" — ISO 8601 일시로 한 번만 실행. 배포 후 검증, 원타임 작업에 적합.

WHAT · 2/3

📡 이벤트 버스 (event.sh)

모든 실행 이력을 JSONL로 기록 — started → completed → delivered 전 과정을 추적 가능하다.

📝

전 과정 기록

각 Job의 시작, 진행, 완료, 전달 상태를 타임스탬프와 함께 JSONL로 저장. 디버깅과 Audit의 SSOT.

🔍

히스토리 조회

event.sh history --limit 10 또는 --job JOB-1234로 특정 Job의 전체 실행 이력 조회 가능.

WHAT · 3/3

🛡️ Watchdog 패턴 (no_agent)

LLM 오버헤드 제로 — 스크립트 stdout이 비어 있으면 silent, 존재하면 알림을 전송한다.

경량 실행 — LLM 추론 없이 순수 스크립트만 실행. 토큰 비용과 지연 시간이 0에 가깝다.

smart-silent — stdout이 비어 있으면 전송 생략. 문제가 있을 때만 알림 → 알림 피로도 최소화.

# Watchdog 설정 예시
no_agent: true
script: "scripts/disk-check.sh"
# stdout 비어있음 → silent
HOW · 1/3

📋 JOB 등록 (action=create)

CLI 명령어 한 줄로 Job을 등록하면 registry.yaml에 자동으로 엔트리가 추가된다.

1
LLM Driven Job

자연어 프롬프트로 복잡한 추론 작업 수행. 매일 아침 뉴스 요약, 보고서 생성 등.

hermes cron create \
  --name "daily-briefing" \
  --prompt "오늘 주요 작업 확인" \
  --schedule "0 9 * * *"
HOW · 2/3

⚡ 스크립트 연결 (script=)

Watchdog 패턴으로 스크립트만 실행. stdout 출력이 있으면 알림 전송, 없으면 silent 처리.

2
Script Driven Job

Wiki 동기화, 데이터 백업, 로그 회전, 시스템 모니터링 등 LLM이 필요 없는 작업.

hermes cron create \
  --name "wiki-sync" \
  --script "scripts/wiki-sync.sh" \
  --schedule "every 5m" \
  --no_agent
HOW · 3/3

📬 배포 채널 (deliver=)

결과를 어디로 보낼지 지정한다 — origin은 시작된 대화 창, all은 모든 연결 채널.

💬

origin

작업이 시작된 대화 창으로 자동 전달. 맥락을 유지한 채 결과 수신.

📢

origin,all

현재 채팅 + 모든 연결 채널 동시 전달. 넓은 범위의 팀 공유용.

🎯

특정 채널

discord:123456 또는 telegram:-100... 식으로 특정 채널 명시 전달.

Pros

✅ 장점

완전 자동화 — 사용자 개입 제로. 설정만 하면 24/7 실행. 에이전트가 자지 않는 시스템.

경량 실행 — no_agent 모드에서 LLM 오버헤드 없음. 스크립트만 실행하여 토큰 비용과 지연 시간 0에 수렴.

병렬 처리 — 독립적인 Job이 동시에 실행 가능. 서로 블로킹 없이 각자의 스케줄에 따라 작업 수행.

Cons

⚠️ 한계

cron 의존성 — system crontab 또는 OpenClaw cron에 의존. 둘 중 하나가 죽으면 전체 Job 스케줄링 중단.

모니터링 필요 — Job 실패 시 자동 알림이 없으면 방치됨. event.sh history를 주기적으로 확인해야 함.

자동 재시도 없음 — 한 번 실패한 Job은 다음 스케줄까지 재실행되지 않음. 수동 개입이 필요할 수 있음.
