## 멀티 에이전트가 필요한 이유

Hermes는 기본적으로 **단일 에이전트**로 작동하지만, 복잡한 작업은 여러 에이전트에게 분산하는 것이 효율적입니다.

### 단일 에이전트의 한계
| 한계 | 설명 |
|------|------|
| **컨텍스트 윈도우** | 하나의 에이전트가 감당할 수 있는 정보량에 한계 |
| **전문성 분리** | 모든 도메인에 동일한 수준의 전문성 적용 |
| **병렬 처리 불가** | 한 번에 하나의 작업만 수행 |
| **회복탄력성** | 에이전트 오류 시 전체 작업 중단 |

## delegate_task — 작업 위임

`delegate_task` 도구를 사용하여 하위 에이전트에 작업을 위임할 수 있습니다.

### 기본 사용법

```
delegate_task(
  task="API 문서를 분석하여 보안 취약점을 찾아줘",
  context={"api_url": "https://api.example.com/docs"}
)
```

### 위임 가능한 작업 유형
- **조사**: 특정 주제에 대한 심층 조사
- **분석**: 코드 리뷰, 보안 감사
- **문서화**: 기술 문서 작성
- **테스트**: 단위 테스트, 통합 테스트
- **모니터링**: 시스템 상태 확인

## Kanban — 시각적 작업 관리

Kanban 시스템을 활용하여 여러 작업을 시각적으로 관리할 수 있습니다.

### Kanban 보드 구조

```
┌─────────┐ ┌─────────┐ ┌─────────┐ ┌─────────┐
│ Backlog │ │ In Prog │ │  Review │ │  Done   │
├─────────┤ ├─────────┤ ├─────────┤ ├─────────┤
│ Task A  │ │ Task B  │ │ Task C  │ │ Task D  │
│ Task E  │ │         │ │         │ │         │
└─────────┘ └─────────┘ └─────────┘ └─────────┘
```

### Kanban 명령어

```
# 보드 보기
/kanban

# 작업 추가
/kanban add "Task: API 문서 업데이트"

# 상태 변경
/kanban move TASK-123 "In Progress"
```

## Codex CLI 위임

Hermes는 OpenAI Codex CLI에 코드 작업을 위임할 수 있습니다.

### Codex CLI 활용 시나리오

```
1. Hermes가 작업 계획 수립
2. Codex CLI에 코드 구현 위임
3. Codex CLI가 코드 작성 및 테스트
4. Hermes가 결과 검토 및 통합
```

### 위임 예제

```
JOB으로 만들어줘:
1. Hermes가 API 명세서 분석
2. Codex CLI에 구현 위임
3. Hermes가 코드 리뷰
4. 최종 통합
```

## 멀티 에이전트 패턴

### 패턴 1: 조사-실행 분리
```
에이전트 A: 조사 담당 (정보 수집, 분석)
에이전트 B: 실행 담당 (코드 작성, 배포)
```

### 패턴 2: 검토 이중화
```
에이전트 A: 코드 작성
에이전트 B: 코드 리뷰 (다른 관점)
```

### 패턴 3: 병렬 처리
```
에이전트 A: 프론트엔드 작업
에이전트 B: 백엔드 작업
에이전트 C: 문서 작업
```

## 주의사항

### ⚠️ 과도한 분할 금지
너무 작은 단위로 작업을 분할하면 오버헤드가 증가합니다. 하나의 에이전트가 처리할 수 있는 작업은 직접 처리하는 것이 효율적입니다.

### ⚠️ 컨텍스트 전달
하위 에이전트에 충분한 컨텍스트를 전달해야 합니다. 너무 적은 정보를 주면 엉뚱한 결과가 나올 수 있습니다.

### ⚠️ 결과 검증
위임된 작업의 결과는 반드시 검증하세요. 멀티 에이전트는 생산성을 높이지만, 품질 관리는 더 중요해집니다.

> **💡 Tip**: 멀티 에이전트는 강력하지만 항상 필요한 것은 아닙니다. 단순한 작업은 단일 에이전트로 충분합니다. 복잡도가 높은 작업에서만 멀티 에이전트를 고려하세요.
