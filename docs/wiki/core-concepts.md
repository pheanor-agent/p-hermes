# 🧠 Core Concepts — Hermes 핵심 개념

> Hermes를 구성하는 5대 개념을 5~10분 안에 이해합니다.
> 각 개념은 독립적이지만 유기적으로 연결됩니다.

---

## 1. Job — Hermes의 작업 단위

**Job**은 Hermes에서 가장 기본적인 작업 단위입니다. 모든 요청은 Job으로 등록되어 추적됩니다.

### Job의 특징
- **추적 가능**: 각 Job은 고유한 ID(`JOB-1234`)를 가집니다
- **상태 관리**: 진행 상황을 단계별로 추적합니다
- **산출물 보존**: 각 단계의 결과물이 자동 저장됩니다
- **재현 가능**: 동일한 Job ID로 전체 이력을 조회할 수 있습니다

### Job의 상태 흐름
```
접수 → 조사 → 설계 → 검토 → 승인 → 실행 → 테스트 → 실행리뷰 → 교훈
```

### Job 기반 사고의 장점
- 작업이 분산되지 않습니다 — 하나의 Job이 끝까지 책임집니다
- 중간에 멈춰도 재개할 수 있습니다 — 상태가 보존됩니다
- 실수해도 롤백 지점이 있습니다 — 각 단계가 체크포인트입니다

## 2. Knowledge — 에이전트의 장기 기억

**Knowledge**는 Hermes의 장기 기억 시스템입니다. 세션이 끝나도 지식은 보존되며, 필요할 때 검색하여 사용합니다.

### 3-Tier 구조

| 계층 | 이름 | 내용 | 예시 |
|------|------|------|------|
| **T1** | 사실 (Facts) | 절대 변하지 않는 정보 | API 엔드포인트, 비밀번호 규칙 |
| **T2** | 맥락 (Context) | 프로젝트 관련 정보 | 아키텍처 결정, 설계 이유 |
| **T3** | 추론 (Reasoning) | 분석 결과, 판단 | 코드 리뷰 결과, 성능 분석 |

### Knowledge의 저장 방식
- **자동 저장**: 대화 내용이 세션 DB에 저장됩니다
- **명시적 저장**: `knowledge` 명령어로 특정 정보를 저장합니다
- **구조화**: 태그, 카테고리, 메타데이터로 분류됩니다

### 검색 방법
- **FTS5 전문 검색**: SQLite 기반의 빠른 전문(full-text) 검색
- **필터 검색**: 태그, 날짜, 계층별 필터링
- **연관 검색**: 관련 Knowledge를 자동 추천

## 3. Skill — 재사용 가능한 절차적 지식

**Skill**은 특정 작업을 수행하는 방법을 정의한 절차적 지식입니다. 한 번 학습한 작업 패턴을 Skill로 저장하면 언제든지 재사용할 수 있습니다.

### Skill의 구성
```
SKILL.md
├── YAML frontmatter (메타데이터)
│   ├── name: 스킬 이름
│   ├── description: 설명
│   └── triggers: 실행 조건
├── Steps: 수행 단계
├── Commands: 구체적인 명령어
└── Pitfalls: 주의사항
```

### Skill의 활용
- **반복 작업 자동화**: 환경 설정, 배포, 코드 리뷰
- **지식 전수**: 팀의 모범 사례를 Skill로 공유
- **품질 표준화**: 모든 작업이 동일한 기준으로 수행

### 예시: "배포 준비" Skill
```yaml
name: deployment-prep
description: 배포 전 준비 작업
steps:
  1. 의존성 확인 (pip list --outdated)
  2. 테스트 실행 (pytest)
  3. 빌드 (python -m build)
  4. 릴리스 노트 작성
```

## 4. Workflow Gate — 9단계 품질 게이트

**Workflow Gate**는 모든 Job이 통과해야 하는 9단계 품질 검증 시스템입니다. 단계를 건너뛸 수 없으며, 각 단계에서 산출물이 검증됩니다.

### 9단계 흐름

| 단계 | 이름 | 산출물 | 검증 항목 |
|:----:|------|--------|----------|
| 1 | **접수** | 요청서 | 요청 명확성 |
| 2 | **조사** | 조사보고서 | 관련 정보 수집 |
| 3 | **설계** | 설계문서 | 접근법 타당성 |
| 4 | **리뷰** | 리뷰결과 | 설계 검토 |
| 5 | **승인** | 승인기록 | 승인 조건 확인 |
| 6 | **실행** | 실행로그 | 코드 변경 |
| 7 | **테스트** | 테스트결과 | 테스트 통과 |
| 8 | **실행리뷰** | 리뷰결과 | 최종 검토 |
| 9 | **교훈** | 회고록 | 개선점 기록 |

### Workflow Gate가 필요한 이유
- **품질 보증**: 모든 작업이 동일한 기준으로 검증됩니다
- **오류 방지**: 조사 없이 실행하거나, 테스트 없이 배포하는 것을 방지합니다
- **감사 가능**: 모든 단계의 의사결정이 기록됩니다
- **회고 가능**: 완료 후 각 단계를 되돌아볼 수 있습니다

## 5. Project — 작업과 지식의 컨테이너

**Project**는 관련 Job과 Knowledge를 담는 컨테이너입니다. 프로젝트 단위로 작업을 그룹화하고, 관련 지식을 함께 관리합니다.

### Project의 구성
```
project-root/
├── project.yaml      # 프로젝트 메타데이터
├── docs/             # 프로젝트 문서
├── specs/            # 설계 명세서 (Spec)
├── src/              # 소스 코드
├── tests/            # 테스트
└── .hermes/          # Hermes 설정
```

### Project의 역할
- **작업 그룹화**: 여러 Job이 하나의 프로젝트에 속할 수 있습니다
- **지식 격리**: 프로젝트별 Knowledge가 분리되어 관리됩니다
- **일관성 유지**: 프로젝트 설정이 모든 Job에 적용됩니다

### Project Lifecycle

```
생성 → 활성 → 유지보수 → 보관 → 종료
```

각 단계에서 관련 Job이 생성되고 Knowledge가 축적됩니다. 프로젝트가 종료되면 모든 산출물이 보관됩니다.

## 개념 관계도

```
사용자 요청
    │
    ▼
┌──────────┐    ┌──────────────┐
│   Job    │───▶│ Workflow Gate│
│ (작업)   │    │ (9단계 품질)  │
└──────────┘    └──────────────┘
    │                    │
    ▼                    ▼
┌──────────┐    ┌──────────────┐
│ Skill    │    │  Knowledge   │
│ (방법)   │    │  (지식)      │
└──────────┘    └──────────────┘
    │                    │
    └────────┬───────────┘
             ▼
    ┌──────────────┐
    │   Project    │
    │ (컨테이너)   │
    └──────────────┘
```

- **Job**이 Workflow Gate를 통과하면서 **Knowledge**를 생성합니다
- **Skill**이 Job의 실행 방법을 제공합니다
- 모든 산출물은 **Project** 단위로 관리됩니다

## 핵심 요약

| 개념 | 한 문장 정의 |
|------|-------------|
| **Job** | 추적 가능한 작업 단위, Hermes의 기본 실행 단위 |
| **Knowledge** | 장기 기억 시스템, 3-Tier 구조로 저장 |
| **Skill** | 재사용 가능한 절차적 지식, 학습한 패턴의 저장소 |
| **Workflow Gate** | 9단계 품질 검증 시스템, 무분별한 실행 방지 |
| **Project** | Job과 Knowledge의 컨테이너, 작업 그룹화 |

> **💡 Tip**: 이 개념들은 모두 연결되어 있습니다. Job을 실행하면 Knowledge가 생기고, Knowledge가 쌓이면 더 좋은 Skill을 만들 수 있습니다.
