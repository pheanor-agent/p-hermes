# 튜토리얼: 첫 프로젝트 만들기

💡 **Hermes와 함께 아이디어를 실제 작동하는 시스템으로 구현하는 전 과정을 실습합니다. '개인 메모 정리 시스템' 구축을 통해 JOB 요청부터 결과 확인까지의 흐름을 익혀보세요.**

## 🌱 기본 개념
이 튜토리얼의 핵심은 **'추상적인 아이디어'를 '구체적인 설계'로 바꾸고, 이를 '실제 코드'로 구현하는 공학적 프로세스**를 경험하는 것입니다.

비유하자면, 여러분은 **'건물주(사용자)'**이고 Hermes는 **'총괄 건축가 겸 시공사(에이전트)'**입니다. 여러분은 \\\"이런 집을 갖고 싶어\\\"라고 말하기만 하면 됩니다. 그러면 Hermes가 땅을 조사하고, 설계도를 그리고, 여러분의 승인을 얻은 뒤 집을 짓습니다. 여러분은 중간에 설계도가 마음에 안 들면 수정 요청을 하고, 최종적으로 지어진 집이 요청한 대로 되었는지 확인만 하면 됩니다. '설계-검토-시공'이라는 전문적인 건축 공정을 함께 밟는 것입니다.

## 🔍 문제 상황: 왜 튜토리얼이 필요한가?
처음 Hermes를 접하면 \"그냥 채팅으로 시키면 되는 거 아냐?\"라고 생각하기 쉽습니다. 하지만 복잡한 시스템을 만들 때 단순히 채팅만으로 진행하면 다음과 같은 문제가 발생합니다:

- **방향성 상실 (Scope Drift)**: 구현 도중 \"어, 내가 원래 원했던 건 이게 아닌데?\"라고 깨닫지만, 이미 코드가 너무 많이 짜여 있어 되돌리기 힘듦. 이는 설계도 없이 집을 짓다가 나중에 방 위치를 바꾸려는 것과 같습니다.
- **검증 누락 (Edge Case Blindness)**: 기능은 구현되었지만, 특수한 상황(예: 인터넷 끊김, 잘못된 입력)에서 시스템이 어떻게 작동하는지 확인하지 않아 배포 후 장애 발생. 이는 지붕의 방수 테스트 없이 완공한 것과 같습니다.
- **인수인계 불가 (Lack of Traceability)**: 나중에 코드를 수정하려 해도, 왜 이런 구조로 짰는지 기록이 없어 다시 분석해야 함. 이는 집의 전기 배선도를 남기지 않고 벽을 마감한 것과 같아 나중에 전등 하나 바꾸려 해도 벽을 다 뜯어야 하는 상황을 만듭니다.

이 튜토리얼에서는 이러한 문제를 해결하는 **'JOB 기반 개발 방식'**을 실습합니다.

## 🏗️ 기술 설계: 실습 프로젝트 - [Obsidian-Discord 메모 시스템]
우리가 만들 시스템은 간단합니다: **\"디스코드에서 특정 키워드와 함께 메시지를 보내면, 그것이 자동으로 내 Obsidian 메모장에 파일로 저장되는 시스템\"**입니다.

### 1. 요구사항 정의
- **입력**: 디스코드 메시지 (예: `!memo 오늘 읽은 책: 클린 코드 - 124p 내용 좋음`)
- **처리**: Hermes가 메시지를 감지 → 키워드(`!memo`) 추출 → 내용 분리 → 파일명 생성(날짜_제목).
- **출력**: Obsidian 보관함(`Vault`) 내의 `Inbox/` 폴더에 `.md` 파일로 저장.

### 2. 구현을 위한 핵심 스킬 조합
이 프로젝트를 위해 Hermes는 내부적으로 다음 스킬들을 조합하여 사용할 것입니다:
- **Discord API 스킬**: Webhook 또는 Bot API를 통한 실시간 메시지 수신 및 필터링.
- **Obsidian/FileSystem 스킬**: OS의 파일 시스템 권한을 활용하여 지정된 절대 경로에 마크다운 파일 생성 및 쓰기.
- **Custom Script 스킬**: 두 서비스를 연결하는 브릿지 스크립트(Python/Node.js)를 작성하고, 이를 백그라운드 프로세스로 실행하여 상시 대기 상태로 만듦.

## 📊 실습 단계별 가이드
이제 실제로 Hermes에게 요청을 보내보세요.

```mermaid
graph LR
    A["💡 아이디어"] --> B["📩 JOB 요청"]
    B --> C["🔍 Investigation"]
    C --> D["📐 Design 설계"]
    D --> E{"✅ 승인?"}
    E -- "예" --> F["⚙️ Execution"]
    E -- "수정 요청" --> D
    F --> G["🧪 Test"]
    G --> H["📋 결과 확인"]
```

### 1단계: 프로젝트 생성 및 JOB 요청
먼저 프로젝트라는 큰 그릇을 만들고, 첫 번째 작업을 요청합니다.
> \" [JOB-0001] '개인 메모 정리 시스템' 프로젝트를 생성하고, 디스코드 메시지를 Obsidian 파일로 저장하는 기능을 구현해줘. 슬러그는 `memo-system`으로 해줘.\"

### 2단계: 설계서(Design) 검토 및 승인
Hermes가 조사를 마치고 `design.md`를 제시할 것입니다. 이때 단순히 \"진행해\"라고 하기보다 다음과 같은 공학적 관점에서 확인하세요:
- **저장 경로**: \"내 Obsidian 보관함 경로가 `/home/bot/obsidian/vault`인데, 여기로 설정되었나?\" (절대 경로 확인)
- **필터링**: \"메시지 내용 중 `!memo`가 없는 일반 채팅은 무시하도록 설계되었나?\" (정규표현식 확인)
- **에러 처리**: \"만약 Obsidian 폴더가 읽기 전용이라면 어떻게 처리할 것인가?\" (Fallback 전략 확인)
**모든 것이 확인되었다면 \"진행해\"라고 답하세요.**

### 3단계: 실행 및 테스트 결과 확인
Hermes가 코드를 작성하고 테스트를 수행합니다. 완료 보고가 오면 직접 확인해 보세요:
- 디스코드에 `!memo 테스트 메모입니다`라고 전송.
- Obsidian 폴더에 `2026-06-15_테스트 메모.md` 파일이 생성되었는지 확인.

## 💡 배운 점 (Key Takeaways)
- **JOB ID의 힘**: `[JOB-0001]`이라는 이름표를 붙임으로써, 이 모든 과정이 하나의 기록으로 남고 나중에 언제든 복기할 수 있게 되었습니다.
- **설계의 중요성**: 코드를 짜기 전 `design.md`를 통해 생각의 오류를 먼저 잡아냈기 때문에, 실행 단계에서 시행착오를 크게 줄였습니다.
- **스킬의 조합**: 에이전트가 가진 여러 전문 지식(Discord, Obsidian)을 적절히 조합하여 새로운 기능을 창조하는 과정을 경험했습니다.
