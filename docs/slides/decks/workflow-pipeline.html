WHY

왜 상태머신인가?

실행의 함정 — 3가지 치명적 사고

① 컨텍스트 미스 (Context Miss)
config.yaml의 들여쓰기를 파악하기 전에 값 하나만 바꾸다가 → 시스템 전체 부팅 실패
② 롤백 불가 (Irreversible Change)
수백 줄 스크립트를 한 번에 재작성 → 기존 예외 처리 로직 증발, 수 시간 복구
③ 에이전트 폭주 (Agent Rampage)
"아키텍처를 효율적으로 바꿔줘" → 15개 파일 동시 수정 → 경로 참조 붕괴, 복구 6시간

근본 원인: 프로세스의 부재

흔한 오해
"더 똑똑한 모델을 쓰면 해결되지 않을까?"
실제 해답
문제는 지능이 아니라 '관문'의 부재. 측정 없는 설계 = 공학적 사고.

→ 상태머신: 유효 전이만 허용하고, 각 단계에 산출물을 강제한다

ATM 비유: 비밀번호 없이 금액 선택으로 점프할 수 없는 것처럼

WHAT

9단계 상태머신

9단계 상태 전이도

점선(피드백 루프): 설계 결함·테스트 실패 시 이전 단계로 되돌아감

각 단계의 역할과 산출물

#단계핵심 목적필수 산출물
1Request요구사항 명확화, 범위 설정request.md
2Investigation현재 시스템 상태·의존성 파악investigation.md
3Design수정할 코드와 구조의 상세 설계design.md (SSOT)
4Review설계서의 논리적 허점 검토review.md
5Approval사용자 최종 승인— (인간 게이트)
6Execution설계서에 기반한 실제 코드 수정execution.log
7Test기능 작동·회귀 테스트test-results.json
8Exec Review실행 결과 ↔ 설계 일치 확인execution-review.md
9Done결과 보고·지식 동기화result.md

→ 단계별 산출물로 추적 가능성(Traceability) 확보

핵심 분리: Investigation → Design → Execution

Investigation
"어떻게 고칠까" 이전에 "지금 어떻게 되어 있는가"를 먼저 본다
Design
코드 한 줄 쓰기 전에 design.md에 모든 변경을 텍스트로 — 이것이 SSOT
Execution
오직 design.md에 적힌 대로만. 임의 판단 금지

무효 전이 금지

Investigation을 건너뛰고 Design으로 가면, 에이전트는 '기존 지식'만으로 설계를 시작한다.

= 측정 없는 설계와 동일한 위험

Approval 단계 — 인간의 최종 승인권

시스템의 가장 핵심적인 안전장치

"진행해" / "승인"이라고 명시하기 전까지
에이전트는 절대 patch·write_file 도구로 파일을 수정하지 않는다
❌ 좋지 않은 승인

"응, 진행해."
(설계 오류를 방치 → 나중에 수정 비용 발생)

✅ 권장되는 승인

"src/utils.py는 새 래퍼 함수로 분리해줘. 그 점만 반영하면 승인할게."

→ 구체적 피드백일수록 Design 단계로 되돌아가 정교화 → 리스크 제거 후 실행

3계층 방어 체계

방어 계층역할
① 구조적 검증
workflow-gate.sh		 상태 전이 검증 — LLM 판단과 무관하게 스크립트 레벨에서 무효 전이 즉시 차단
② 콘텐츠 검증
Content System		 content: true 시 L1(구조)→L2(오류)→L3(표현)→L4(도메인), Blog 도메인은 L5 Judge 추가
③ 인간 승인
Approval Gate		 AI 설계서를 사람이 직접 검토하는 '인지적 허들' — 12% JOB 거부, 평균 3시간 복구 절약

단일 검증이 놓친 문제를 다음 계층이 캐치

HOW

어떻게 작동하는가?

workflow-gate.sh & .workflow-state

상태 확인

$ bash core/scripts/workflow-gate.sh \
      --status --job JOB-2047
{
  "job_id": "JOB-2047",
  "current_step": "design",
  "completed_artifacts": [
    "request.md",
    "investigation.md",
    "design.md"
  ],
  "next_allowed_steps": ["review"]
}

무효 전이 차단

$ workflow-gate.sh --next execution --job JOB-2047
[ERROR] Cannot transition from Design to Execution.
[ERROR] Missing required artifact: review.md
[ERROR] Please complete Review step first.
exit 1

flock 기반 원자적 갱신 — 병렬 실행·인터럽트에도 상태 파일 무결성 보장

실제 사례: JOB-2047 세션 DB 마이그레이션

Investigation — 1,247개 세션, 48MB, 의존 스크립트 3개(sessions.db 직접 조회) 발견

Design — 이중 쓰기 레이어 → 배치 변환 → 스크립트 수정 → 검증 → 아카이브 (5 Phase)

사건 1 (Design) — healthcheck.sh의 PRAGMA integrity_check 의존성 사전 발견. 단순 실행했다면 누락.

사건 2 (Review) — Design→Execution 점프 시도를 workflow-gate.sh가 차단

Execution — 1247/1247 변환(23초), 의존 스크립트 3개 수정

Test — 4/4 PASS · Exec Review — 설계 ↔ 결과 일치 확인

→ 재작업 0회, 롤백 0회로 통제된 마이그레이션 완료

도입 전후 비교 (89개 JOB, 2026년 1-6월)

2.7 → 0.4
재작업 회수/JOB		 47% → 7%
롤백 발생률		 5 → 0
폭주 사고 건수

초기 2주는 학습 기간(72분) → 이후 JOB당 52분, 총 완료 시간은 단축

작업당 비용 32% 절감 · 승인 거부 12%는 평균 3시간 복구 절약 효과

TRADE-OFFS

9단계가 항상 정답은 아니다

TASK vs JOB — 복잡도에 따른 프로세스

유형예시프로세스소요
TASK 파일 한 줄 수정, 간단한 질문 Request → Execution → Done (3단계) 2-5분
JOB 아키텍처 변경, 새 기능 도입 9단계 전체 30-90분
JOB 판단 기준
파일 변경 4개 이상 · 다중 모듈 영향 · 롤백에 별도 계획 필요

향후: 복잡도 평가로 3~9단계 탄력 조정, 저위험 작업은 자동 Approval

핵심 요약

  1. 에이전트 사고의 원인은 지능이 아니라 관문의 부재
  2. 9단계 상태머신: 조사 → 설계 → 검토 → 승인 → 실행 → 검증
  3. Approval 단계가 인간의 통제권을 확보
  4. workflow-gate.sh가 LLM 판단과 무관하게 무효 전이 차단
  5. TASK/JOB 분리로 안전과 속도의 균형
