WHY

왜 명세서가 먼저인가?

명세서 없이 코드부터 작성하면

① 재작업 비용 폭발
방향이 어긋나면 처음부터 다시. 여러 에이전트가 각기 다른 기준으로 작업.
② 수치·구조 불일치
부모"1,500자" vs 자식"3,500자" — 어느 것이 진실? 폴더 경로 재정의도 참조 무효화.
③ 추적 불가
누가, 언제, 왜 변경했는지 알 수 없으면 승인된 산출물조차 시스템과 불일치.
WHAT

Spec-Driven Dev의 핵심 메커니즘

SSOT: 단일 진실 공급원

명세서(Spec)가 시스템의 권위 있는 정의. 모든 것은 Spec에서 파생됩니다.

Spec이 곧 진실
코드 · 테스트 · 문서는 모두 Spec에서 파생
변경은 Spec에서 시작
기능 추가 · 버그 수정 · 리팩토링 모두 Spec 변경으로 시작

-> 자동화된 검증(spec-conformance.sh)으로 준수 여부를 기계가 확인

3종 의존성 타입

타입의미승인부모 업데이트
extends 부모를 상세화하여 확장 ✅ 가능 ❌ 불필요
raises 부모의 수치적 기준 상승 (분량·성능 등) ✅ 가능 (delta 문서화) ✅ 필수
conflicts 부모와 모순 (구조·경로 재정의 등) ❌ 차단 충돌 해결 필요

raises는 충돌이 아닌 의도적 요구사항 상승 — delta를 문서화하고 승인 시 부모를 업데이트

의존성 그래프 예시

  P["SPEC-PARENT\n(부모 Spec)"] -->|"extends"| C1["SPEC-CHILD-1\n(자식 1)"]
  P -->|"extends"| C2["SPEC-CHILD-2\n(자식 2)"]
  P -->|"raises"| C3["SPEC-CHILD-3\n(자식 3)"]
  C1 -->|"conflicts"| C2

부모는 구조/권위, 자식은 부분집합 상세화 — 계층 구조 유지

Conflict Detection: 3단계 검증

① Quantitative
수치 비교		 분량·라인 수·성능 지표 비교. 상승 -> raises, 모순 -> conflicts
② Structural
구조 재정의		 폴더 구조·트랙 아키텍처·파일 경로 재정의 시도 -> conflicts (경로 무효화 방지)
③ Domain Overlap
도메인 중복		 동일 도메인 두 Spec의 역할 중복 확인. 부모=구조, 자식=부분 상세화

-> 신규 Spec 등록 전 반드시 3단계를 모두 통과해야 승인

검증 흐름도

graph TD
  NEW["New Spec\n신규 명세서"] --> Q["Step 1: Quantitative\n수치 비교"]
  Q -->|"일치"| S["Step 2: Structural\n구조 검증"]
  Q -->|"모순"| REJ["❌ Rejected\nconflicts"]
  S -->|"일치"| D["Step 3: Domain Overlap\n도메인 중복 확인"]
  S -->|"모순"| REJ
  D -->|"일치"| OK["✅ Approved\n승인"]
  D -->|"중복"| REJ

HOW

어떻게 작동하는가?

Batch Modification: 병렬 수정

리뷰에서 여러 Spec 수정 사항이 발견되면 서브에이전트에게 병렬로 위임

graph TD
  REV["Review\n리뷰 단계"] -->|"수정 사항 N개"| SPC["Spec Coordinator\n조정자"]
  SPC --> SA1["Sub-Agent 1\nSpec A 수정"]
  SPC --> SA2["Sub-Agent 2\nSpec B 수정"]
  SPC --> SA3["Sub-Agent N\nSpec N 수정"]
  SA1 --> VG["Validation\n검증"]
  SA2 --> VG
  SA3 --> VG
  VG -->|"PASS"| MER["Merge\n병합"]
  VG -->|"FAIL"| FIX["↩ 재수정"]

TRADE-OFFS 한계와 주의점

장점 비용
변경 추적성 · 일관성 확보 초기 Spec 작성 오버헤드
다중 에이전트 충돌 사전 차단 검증 절차(grep/체크리스트) 학습 곡선
코드·문서·테스트의 자동 검증 작은/일회성 프로젝트엔 과잉
이런 경우엔 부적합
단일 파일 스크립트 · 빠른 프로토타입 · Spec 간 관계가 없는 독립 작업 — 절차 비용만 커짐

핵심 요약

  1. SSOT — 모든 코드·문서·테스트는 Spec에서 파생
  2. 3종 의존성 — extends(확장) / raises(상승) / conflicts(충돌)
  3. 3단계 Conflict Detection — 수치 -> 구조 -> 도메인, 등록 전 필수
  4. Batch Modification — 병렬 위임 + patch + 검증으로 다중 Spec 수정
