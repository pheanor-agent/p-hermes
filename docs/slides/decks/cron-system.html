WHY

왜 자동화가 필요한가?

반복 작업이 수동일 때 겪는 문제

① 에이전트는 '자'는 시간이 있음
세션이 종료되면 모든 작업 정지. 매일 아침 뉴스 요약 같은 루틴은 사용자가 직접 매번 요청해야 함.
② 사람이 기억하면 누락 생김
주간 점검, 데이터 백업, 로그 회전 — 기억에 의존하면 일정 충돌과 중복 실행이 필연적.
③ 실패해도 아무도 모름
수동으로 실행한 작업이 실패하면 재시도도, 알림도 없음. 문제를 지나치게 늦게 발견.

registry.yaml — 모든 Job의 단일 진실 출처

~/.hermes/cron/registry.yaml


- id: morning-news
  name: "매일 아침 뉴스 요약"
  schedule: "0 8 * * *"
  prompt: |
    오늘 AI/기술 분야 주요 뉴스 5개를 검색하고
    각 항목별로 2문장 이내로 요약해 주세요.
  deliver: "discord:123456789"
  model:
    provider: openrouter
    model: "anthropic/claude-sonnet-4"
  enabled_toolsets: ["web", "terminal"]

- id: deploy-check
  name: "배포 후 상태 점검"
  schedule: "2026-07-01T14:30:00"
  prompt: "서버 상태를 확인하고 응답 시간을 기록하세요."
  deliver: "discord:123456789"

→ ISO 8601 일시면 one-shot 단회 실행, cron 표현식면 주기적 반복

event.sh — 실행 이력의 중앙 기록소

이벤트 타입
startedJob 시작
running추론 진행 중
completed작업 완료
delivered결과 배송됨
JSONL 로그 형식 
{
  "ts": "2026-06-17T08:00:01Z",
  "job": "morning-news",
  "event": "started",
  "session": "sess-abc123"
}
{
  "ts": "2026-06-17T08:02:15Z",
  "event": "completed",
  "result": "success"
}

event.sh history --limit 10 · event.sh history --job "JOB-1234"

3가지 Job 타입 — 작업 성격에 맞는 실행 방식

🧠 LLM-Driven Job — 자연어 프롬프트로 복잡한 작업 수행
예: 작업 리포트 생성, 데이터 분석 요약, 코드 리뷰
hermes cron create --name "daily-report" --prompt "..." --schedule "0 9 * * *"
⚡ Script-Driven Job — LLM 오버헤드 없이 스크립트만 실행
예: Wiki 동기화, 데이터 백업, 로그 회전, 시스템 모니터링
hermes cron create --name "wiki-sync" --script "scripts/wiki-sync.sh" --schedule "every 5m"
🔔 Event-Driven Job — 특정 이벤트 발생 시 즉시 실행
예: GitHub PR 생성 시 리뷰, Slack 메시지 응답, 웹훅 처리
hermes cron create --name "pr-review" --trigger "github.pr.created" --prompt "..."

silent-on-success & origin — 지능형 예외 처리

🤫 silent-on-success
정상 실행 시 알림 전송 안 함. 실패/경고 시만 결과 전달 → 알림 피로도 낮춤.

no_agent: true + script
스크립트 출력이 비어 있으면 전송 생략, 존재하면 알림 생성.
📮 origin 채널 해석
deliver: origin — 작업이 시작된 대화 창으로 자동 전달.
deliver: origin,all — 현재 채팅 + 모든 연결 채널 동시 전달.
deliver: "discord:123" — 특정 채널 명시 전달.

다중 채널 전달 시 실패 시 fallback 자동 동작.
HOW

실제로 어떻게 작동하는가?

실제 CLI 명령어로 Job 등록하기


# LLM 구동 — 매일 9시 작업 현황 리포트
hermes cron create \
  --name "daily-briefing" \
  --prompt "오늘의 주요 작업 확인" \
  --schedule "0 9 * * *" \
  --deliver telegram

# 스크립트 전용 — 5분마다 Wiki 동기화
hermes cron create \
  --name "wiki-sync" \
  --script "scripts/wiki-sync.sh" \
  --schedule "every 5m"

# 이벤트 기반 — PR 생성 시 자동 리뷰
hermes cron create \
  --name "pr-review" \
  --trigger "github.pr.created" \
  --prompt "PR 코드 리뷰 수행" \
  --deliver discord

등록 시 registry.yaml에 자동으로 YAML 엔트리 추가 → 스케줄러가 감지

실행 시 이벤트 흐름 — end-to-end

sequenceDiagram participant S as Scheduler participant E as Event Bus participant W as Wrapper participant R as Runner participant D as Delivery Note over S,D: 스케줄 도달 S->>E: 스케줄 도달 E->>W: publish "started" W->>W: .env 로드 / 세션 생성 W->>R: 매개변수 전달 E->>R: publish "running" R->>R: 에이전트 추론 R->>D: 결과 전달 E->>R: publish "completed" D->>D: Discord/Telegram 전송 E->>D: publish "delivered"

모든 단계가 event.sh에 JSONL로 기록 → event.sh history로 전체 추적 가능

자율적 데이터 수집 / 변경 감지 패턴

📊 주기적 모니터링
15분마다 시스템 상태 확인 → 이상치 발생 시만 알림 (silent-on-success)

🔍 변경 감지
mutex_acquire로 중복 실행 차단
스크립트 출력이 비어 있으면 '변경 없음'으로 처리

🔄 하이브리드 패턴
시간 기반 + 이벤트 기반 조합
예: 매주 월요일 + GitHub 이벤트 발생 시 동시 실행
system-common 유틸리티
mutex_acquire중복 실행 차단
mkdir_atomic원자적 디렉토리 생성
log_event표준 이벤트 기록

mutex_acquire "morning-news" || exit 0
# ... 작업 실행 ...
mutex_release "morning-news"
TRADE-OFFS

무인 실행의 리스크

자동화에서 주의할 점

⚠️ 잘못된 Job의 연쇄 효과
Event-Driven Job이 잘못 해석된 이벤트를 연쇄적으로 처리하면 오류가 기하급수적으로 증폭됨.
⚠️ LLM 출력의 비결정성
같은 프롬프트라도 매번 다른 결과가 나올 수 있음. 스크립트 기반 Job은 확정적이지 않음.
⚠️ 도구 권한 남용 가능성
enabled_toolsets를 제한하지 않으면 cron Job이 의도치 않은 파일 수정이나 API 호출을 함.

→ countermeasures: silent-on-success, mutex_acquire, enabled_toolsets 제한, event.sh history 모니터링

핵심 요약

  1. registry.yaml이 모든 Job 정의의 SSOT — YAML로 작업·스케줄·전달 채널 정의
  2. 3층 아키텍처로 역할 분리 — Scheduler → Wrapper → Runner
  3. event.sh가 모든 실행 이력을 JSONL로 기록 — 추적/디버깅/Audit 가능
  4. 3가지 Job 타입 — LLM 구동, 스크립트 전용, 이벤트 기반 중 작업 성격에 맞춤
  5. silent-on-success + origin 채널 — 지능형 알림으로 피로도 최소화
