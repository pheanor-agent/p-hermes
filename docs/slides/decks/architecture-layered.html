WHY

왜 레이어드 구조인가?

모놀리식의 3가지 한계

① 파일 간 의존성 불명확
모든 스크립트가 한 디렉토리에 혼재. 한 파일 수정이 예상치 못한 곳에서 오류 발생.
② 절대 경로 하드코딩
WSL ↔ macOS 간 이동 시 전체 경로 재수정 필요. 환경마다 다른 설정 파일.
③ 직접 호출 = 결합도 폭발
모듈 A가 모듈 B를 직접 호출하면 교착 상태(deadlock)와 종속성 지옥.

패러다임 전환

Flat (모놀리식)
Layered (레이어드)

직접 호출 → 이벤트 기반 비동기 통신으로 결합도 해방

WHAT

관계망 아키텍처

5계층 + 2글로벌 = 관계망

단방향 직선이 아닌, 실제 의존 관계 기반 망 구조

레이어별 역할

계층주요 구성책임
T1 Core config.yaml, AGENTS.md, skills/, workflow.sh 워크플로우 엔진, 스킬 정의, 전역 설정(SSOT)
T2 Runtime knowledge/, cron/, sessions/, .workflow-state 실행 시간 데이터, 지식, 자동화, 세션 상태
T3 Interfaces discord/, telegram/, CLI 외부 통신 채널 (프로토콜 차이는 계층 내부 처리)
T4 Infra GitHub, Docker, WSL/Host 실행 환경, 지속화, CI/CD
T5 Release docs/, slides/, llms.txt 배포 산출물, 문서, LLM 메타데이터

3대 설계 원칙

① 심링크 금지
물리적 경로만 사용. 심링크는 환경 간 경계에서 파괴됨. readlink -f로 절대 경로 확보.
② $HERMES_ROOT 추상화
절대 경로 하드코딩 금지. 모든 경로 = 환경 변수 기반. WSL ↔ macOS 호환.
③ 이벤트 기반 통신
모듈 간 직접 호출 금지. event.sh Pub/Sub 패턴으로 비동기 상태 전달.

event.sh — 시스템의 신경망

모든 모듈은 event.sh를 통해 통신. 직접 호출 X, 이벤트 발행/구독 O

이벤트 발행
event.sh publish workflow.started
이벤트 구독
event.sh subscribe workflow.completed

전통적 계층 vs Hermes 관계망

전통적 5계층Hermes 관계망
통신 방식 인접 계층만 ↔ 호출 event.sh로 전 계층 비동기
유틸리티 각 계층별 별도 system-common/ 전역 공유
경로 관리 상대 경로 혼합 $HERMES_ROOT 단일 기준
설정 계층별 중복 config.yaml / AGENTS.md SSOT
확장 전 계층 수정 필요 새 계층 event.sh 연결만
HOW

실제 디렉토리 구조

$HERMES_ROOT 디렉토리 맵

실제 데이터 흐름: Cron 작업 예시

1 Cron이 registry.yaml에서 스케줄 확인
2 event.sh에 cron.job.started 이벤트 발행
3 Core가 이벤트를 구독 → workflow.sh에서 9-Step 실행
4 system-common/lib/atomic.sh로 원자적 상태 파일 작성
5 결과를 session/에 기록, Discord/Telegram으로 전송
6 event.sh에 cron.job.completed 발행 → JSONL 히스토리에 기록

→ 모든 단계가 event.sh를 경유. 직접 호출은 0회.

TRADE-OFFS

한계와 주의점

레이어드 구조의 트레이드오프

장점 비용
계층 간 결합도 ↓, 독립 배포 이벤트 버스 복잡성 ↑, 디버깅 간접적
환경 독립성 (WSL ↔ macOS) $HERMES_ROOT 설정 전제, 초기화 오버헤드
수평 확장 용이 (새 인터페이스 추가 시 T3만 수정) 이벤트 스키마 관리 필요, 버전 마이그레이션
안티패턴 금지로 안정성 ↑ 학습 곡선 — 직접 호출이 습관인 팀엔 부적합

"구조의 자유는 구조의 책임을 전제로 한다"

핵심 요약

  1. 5계층(T1~T5) + 2글로벌(event.sh, system-common) 관계망 구조
  2. 3대 설계 원칙: 심링크 금지, $HERMES_ROOT 추상화, 이벤트 기반 통신
  3. 모듈 간 직접 호출 0회 — event.sh가 유일한 통신 채널
  4. config.yaml + AGENTS.md = SSOT (중복 설정 금지)

"단방향 직선이 아닌, 실제 의존 관계를 반영한 관계망"
