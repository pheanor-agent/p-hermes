WHY

왜 역할 기반 라우팅인가?

단일 모델의 3가지 치명적 한계

① 비용 폭주 (Cost Explosion)
단순 파일 복사에도 토큰당 가장 비싼 모델을 호출. 월 API 예산 순식간에 초과.
② 성능 불일치 (Performance Mismatch)
추론 특화 모델이 단순 코딩에서 '너무 많이 생각' → 시간 40% 증가, 오버엔지니어링.
③ 자기 확증 편향 (Self-Confirmation Bias)
동일 모델이 설계도 하고 검증도 하면 자신의 오류를 감지하지 못함. 설계 결함 발견율 31%에 그침.

만능 모델의 환상 vs 현실

비유: 건축 프로젝트

설계도를 그리는 건축가와 벽돌을 쌓는 숙련공, 현장을 파악하는 조사원이 필요합니다.

건축가에게 벽돌을 쌓게 하면? 효율과 품질 모두 떨어짐.

LLM의 현실 
평균 지능:  추론형 > 범용형
코드 생성:  수행형 > 추론형
논리 추론:  추론형 > 수행형
빠른 응답:  범용형 > 수행형
비용 효율:  범용형 > 추론형

벤치마크는 '평균'만 보여줌.
'특정 작업'에서의 상대적 강약은 다름.

WHAT

역할 기반 라우팅이란?

핵심 아이디어

역할 기반 라우팅 (Role-Based Routing)
작업의 성격(설계, 실행, 검증, 조사)에 따라 가장 적합한 모델을 자동으로 배정합니다. 비용은 낮추고 성능은 극대화하는 '적재적소' 전략입니다.

설정 파일 model-roles.yaml에서 역할→모델 매핑을 정의하고,
라우팅 엔진이 실행 시점에 작업 단계별로 최적 모델을 선택합니다.

4가지 도메인 역할

역할핵심 역량모델 특성주요 작업
추론형 논리적 추론, 창의적 설계 높은 MMLU, 복잡한 지시 이행 아키텍처 설계, design.md 작성
수행형 구문 정확성, 구현 속도 높은 HumanEval, 간결한 코드 파일 수정, 스크립트 구현
검증형 세밀한 검증, 오류 탐지 비판적 사고, 엣지 케이스 발견 설계 검토, 코드 리뷰
범용형 빠른 응답, 폭넓은 검색 낮은 지연시간, 효율적 토큰 사용 파일 구조 분석, 현황 파악

작업 → 역할 매핑

작업 단계 할당 역할 이유
Investigation (조사)범용형빠른 상황 파악이 핵심, 비용 절감
Design (설계)추론형복잡한 논리 구성과 아키텍처 사고
Review (검증)검증형설계 모델과 달라야 함 (교차 검증)
Execution (구현)수행형구문 정확도와 구현 속도
Test (테스트)범용형결과 확인용, 빠른 처리 필요

교차 검증 원칙 (Cross-Check)

❌ 동일 모델 검증
설계와 검증을 같은 모델이 하면?
자기 논리적 오류 감지 불가.
결함 발견율: 31%
✅ 서로 다른 모델 검증
다른 학습 데이터셋을 가진 모델이
완전히 다른 관점에서 검토.
결함 발견율: 78% (+152%)

검증형 API 장애 시 추론형으로 대체 (Fallback)

설정 구조: model-roles.yaml

# SSOT: 모든 역할→모델 매핑의 단일 진리원
model:
  default: general        # 기본 역할
  roles:
    design:
      primary:
        role: reasoning
        description: "논리적 설계 및 아키텍처"
      fallback:
        role: review

    review:
      primary:
        role: review
        description: "비판적 검증 및 코드 리뷰"
      fallback:
        role: reasoning

    execution:
      primary:
        role: execution
        description: "코드 구현 및 파일 수정"

    investigation:
      primary:
        role: general
        description: "빠른 현황 분석 및 검색"

역할 이름만 지정 → 실제 프로바이더/모델명은 infra 레이어에서 해석

HOW

실제 라우팅 흐름

전체 라우팅 파이프라인

워크플로우 단계별 모델 할당 예시

단계모델 역할작업 내용비용 (예시)
1. Investigation 범용형 파일 구조 분석, 시스템 현황 파악 $1.20
2. Design 추론형 상세 설계서 작성, 아키텍처 결정 $12.00
3. Review 검증형 설계 결함 검증, 수정 요청 $15.00
4. Execution 수행형 코드 구현, 파일 수정 $5.50
5. Test 범용형 테스트 실행, 결과 검증 $1.80
합계 $35.50

단일 모델 사용 시 $94.50 대비 62% 절감

TRADE-OFFS

트레이드오프

라우팅의 이득 vs 비용

이득 비용
API 비용 62% 절감 model-roles.yaml + 라우팅 엔진 유지관리
작업 시간 54% 단축 다수 모델 API 통합/모니터링 필요
결함 발견율 3배 향상 (교차 검증) 검증형 모델 사용으로 부분적 비용 재상승
롤백률 82% 감소 단일 모델에 비해 시스템 복잡도 증가

결론: 작업량이 많고 단계별 특성이 뚜렷한 프로젝트에서 이득이 압도적

핵심 요약

  1. 단일 모델으로 모든 작업을 하면 비용 폭주, 성능 불일치, 자기 확증 편향이 발생
  2. 4가지 도메인 역할(추론형/수행형/검증형/범용형)로 작업을 특화 모델에 배정
  3. 교차 검증: Design과 Review를 다른 모델로 → 결함 발견율 31% → 78%
  4. SSOT은 model-roles.yaml. 역할 이름만 정의, 실제 모델명은 infra 레이어 해석
