WHY

왜 지식 시스템인가?

지식이 없으면 세 가지 문제

① 컨텍스트 오버플로우
50개 세션 이력 + 100개 뉴스 요약이 컨텍스트를 가득 채우면,"지금 이 코드를 수정해줘"라는 핵심 지시가 뒤로 밀려남.
② 추론 오염 (Inference Contamination)
6개월 전"A 모델이 최고다"라는 기록을 읽고 최신 벤치마크 결과를 무시. 노후 데이터를 사실로 착각.
③ 검색 불가
grep -rl"모델"로 3가지 다른 맥락이 뒤섞인 결과. 도메인/태그 기반 정밀 검색이 필요.

수치로 본 지식 손실

지표Raw 컨텍스트Wiki DB개선율
사실 기반 응답 정확도 59% 92% +33pp
환각(Hallucination) 발생률 41% 8% -80%
노후 데이터 참조율 28% 3% -89%
컨텍스트 토큰 사용량 32K tokens 4.2K tokens -87%

Raw Data -> 정제된 Wiki로 전환할 때 4개 지표 모두 30~89% 개선

하나는 읽기, 하나는 쓰기 — 원본은 만지지 않고 Wiki만 가공

가공 파이프라인 5단계

단계스크립트역할
① process daily-knowledge-process.sh 변경된 원본 스캔 -> wiki entity/concept 페이지 생성
② clean clean-wiki-pages.sh pages/에 낀 원본 복사본 정리 (인플레이션 방지)
③ graph build-metadata.sh -> build-graph.sh metadata.json + graph_edges.json (페이지 간 관계 그래프)
④ score build-scores.sh scores.json + index.md (T1/T2/T3 분류)
⑤ health knowledge-health-report.sh 일일 건강도 리포트 -> T3 아카이빙 트리거

T1/T2/T3 스코어링 시스템

score = pw × 0.5 + rs × 0.3 + us × 0.2
Tier명칭점수노출 전략
T1 Core (핵심) ≥ 0.70 항상 노출 — 에이전트가 index.md에서 첫 번째로 읽음
T2 Working (작업용) 0.40 ~ 0.69 도메인 컨텍스트에 따라 조건부 노출
T3 Reference (참고) < 0.40 명시적 요청 시만 — 오래되면 아카이빙 대상

Core Immunity: priority: core 지정 시 rs=1.0 고정 -> 시간 경과에도 T1 유지

점수 계산 상세 — pw, rs, us

pw (priority weight) — 우선순위 가중치

  • core = 1.0
  • working = 0.6
  • reference = 0.2

rs (recency score) — 최근성 점수

  • 0일 경과 = 1.0
  • 7일 경과 = 0.8
  • 30일 경과 = 0.5
  • 90일 경과 = 0.3

us (usage score) — 사용량 점수

  • usage.json에서 읽은 실제 호출 횟수
  • 정규화: usage_count / max_usage
  • 자주 참조되는 지식 = 높은 점수
실제 예시
system/models.md -> pw=1.0, rs=0.8, us=0.9
score = 1.0×0.5 + 0.8×0.3 + 0.9×0.2 = 0.74 (T1)

TRADE-OFFS

유지보수 비용과 지식 부패

지식 시스템의 트레이드오프

장점 비용
환각률 41% -> 8% (80% 감소) LLM 호출 비용 (가공 파이프라인)
Token 사용량 -87% (32K -> 4.2K) 점수 재계산 + 그래프 동기화 오버헤드
400배 빠른 정밀 검색 부패한 지식 관리 필요 (wiki-cleanup.sh)
의견/사실 자동 분리 LLM 추출 정확도 의존 (confidence < 0.85 경고)
부패 방지 3가지 규칙
① 1년 경과 + 신뢰도 0.7 미만 -> 아카이브   ② 반증된 정보 -> 즉시 삭제   ③ Core Immunity로 핵심 지식 보호

핵심 요약

  1. Raw 원본은 수정 금지 — append-only, 직접 참조만
  2. LLM 파이프라인이"의견"을 걸러내고"사실"만 Wiki로 정제
  3. T1/T2/T3 스코어링으로 에이전트가 중요한 지식부터 읽음
  4. FTS5 검색으로 3ms 내 정밀 탐색 (grep 대비 400배)
  5. 매일 자동 갱신 — 에이전트가 오래된 지식으로 실수하지 않음

"Raw 데이터는 흙, 파이프라인은 체, Wiki는 정제된 금 — 에이전트는 금만 읽는다"
