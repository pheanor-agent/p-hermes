# Creative Content System Design

> **정체성**: 연출 (연출)
> **대상**: 의미 (의미)

창의적/표현적 콘텐츠 생성을 구조화하는 메타 설계 원칙. 개별 도메인 스킬(seminar-slides, novel-writing, baoyu-comic, baoyu-infographic 등)이 공유하는 패턴과 설계 기준을 정의한다.

---

## 분류 기준: "단순 요약이 아닌, 이해와 흥미를 전달하는 콘텐츠"

콘텐츠를 다음과 같이 분류한다. 핵심 기준은 **독자에게 가치를 전달하는 표현이 필요한가**이다.

### 포함 도메인 (연출 필요)

| 도메인 | 예시 | 공유하는 연출 패턴 |
|--------|------|-------------------|
| **교육/설명** | README, Wiki, 블로그, 가이드 | 계층화, 비유, 어조 적응, 예시 |
| **내러티브/스토리** | 소설, 에세이, 시나리오 | 구조, 어조, 묘사, 설정 일치성 |
| **시각 콘텐츠** | 인포그래픽, 만화, 다이어그램, 일러스트 | 스타일×레이아웃, 데이터 시각화 |
| **프레젠테이션** | 세미나 슬라이드, 대시보드 | 계층화, 비유, 팝업 심화 |
| **이미지** | ComfyUI, 프롬프트 | 프롬프트 엔지니어링, LoRA 조합 |

### 제외 도메인 (단순 기록/자동화)

| 제외 | 이유 |
|------|------|
| Changelog | 변경 이력 로깅, 표현력 불필요 |
| JOB 산출물 요약 | 작업 결과 기록, 구조화만 필요 |
| API 레퍼런스 | 엔드포인트 나열, 사실 전달 |
| Config 설명 | 키-값 설명, 비유 불필요 |

**사용자 지적 **(JOB-1593) "README, Wiki, 블로그도 단순히 요약하는 게 아니라 흥미롭고 이해에 도움이 되는 결과물이 나와야 한다"

## 공유 "연출 엔진" 5가지 구성 요소

모든 포함 도메인이 공유하는 콘텐츠 작성 패턴:

### 1. 계층화 (Tier Generator)

`L1 인상 → L2 개요 → L3 상세` 3층 구조로 콘텐츠 구성. Progressive Disclosure.

| 도메인 | 구현 |
|--------|------|
| README | Overview → Quick Start → 상세 가이드 |
| Wiki | 개념 설명 → 상세 → 예시 |
| 블로그 | 문제 → 인사이트 → 해결책 |
| 소설 | 화 개요 → 절(장면) → 문장(묘사) |
| 슬라이드 | 본문(개념) → 💡Tip → 팝업(기술적 심화) |
| 인포그래픽 | 제목(한 줄) → 라벨(키워드) → 본문(데이터) |

### 2. 어조 적응 (Tone Adapter)

타겟/상황에 따라 어조, 어휘, 비유 방식을 적응.

| 도메인 | 어조 차원 |
|--------|-----------|
| README/Wiki | 기술적 독자 ↔ 비기술적 독자 |
| 블로그 | 캐주얼 ↔ 전문적 |
| 소설 | 캐릭터별 어조 |
| 슬라이드 | 비기술적(본문) ↔ 기술적(팝업) |
| 인포그래픽 | 21가지 스타일 정의 |
| 만화 | 7가지 톤(dramatic, warm, action 등) |

### 3. 비유 생성 (Analogy Builder)

추상 개념 → 구체적인 비유/시각화/행동 변환.

| 도메인 | 비유 패턴 |
|--------|-----------|
| README | "Docker = 컨테이너 박스" |
| Wiki | "지식 시스템 = 도서관 카탈로그" |
| 블로그 | 스토리텔링, 개인 경험 |
| 소설 | 추상 감정 → 신체적/환경적 묘사 |
| 슬라이드 | 💡Tip: "RAG = 도서관 카탈로그 시스템처럼" |
| 인포그래픽 | 데이터 → 시각 메타포(iceberg, funnel, bridge) |

### 4. 템플릿 채움 (Template Filler)

도메인별 템플릿 + LLM 내용 생성. 반복 구성 요소 재사용.

| 도메인 | 템플릿 단위 |
|--------|-------------|
| README | Quick Start, FAQ, Contributing 섹션 |
| Wiki | 도메인, 엔티티, 개념 페이지 |
| 블로그 | 포스트 구조(문제→인사이트→해결) |
| 소설 | 화(2,500-4,000자), 절(800-1,500자) |
| 슬라이드 | 8개 레이아웃 템플릿 |
| 인포그래픽 | 21개 레이아웃 |

### 5. 검증 (Validator)

공통 검증 + 도메인별 검증.

| 검증 항목 | README | Wiki | 블로그 | 소설 | 슬라이드 | 인포 |
|-----------|--------|------|--------|------|---------|------|
| 문자셋(한/중/일) | ✅ | ✅ | ✅ | ✅ | ✅ | ✅ |
| 일관성 | ✅ 링크 | ✅ 도메인/태그 | ✅ 논리 | ✅ 설정 | ✅ 패널 중복 | ✅ 데이터 |
| 분량 | ❌ | ❌ | ❌ | ✅ | ✅ 400자/슬라이드 | ❌ |
| 도메인별 | 명령어 실행 | frontmatter | 문법 | 분량/설정 | 브레이스 균형 | 데이터 정확성 |

## 공유 인프라 (Infra Layer)

연출 엔진이 작동하기 위한 하위 인프라:

### 스타일×매트릭스 시스템

도메인별 스타일/레이아웃/톤 조합을 구조화.

| 기존 매트릭스 | 조합 수 |
|---------------|---------|
| baoyu-comic: 6 아티스트 × 7 톤 × 5 프리셋 | 42+ |
| baoyu-infographic: 21 레이아웃 × 21 스타일 | 441 |
| seminar-slides: 8 레이아웃 템플릿 | 8 |

**공유화 방향**: `style-matrix.json` 공통 정의 + 도메인별 오버레이

### 디자인 토큰

공통 CSS 변수로 시각적 일관성 유지.

```
tokens.css       ← 공통: 폰트, 색상, 간격, 테두리, 그림자
seminar.css      ← 오버레이: 56px 폰트, 흰 배경
technical.css    ← 오버레이: 다크 코드 블록, 모노스페이스
```

### 콘텐츠 분석 파이프라인

원본 콘텐츠 → 분석 → 구조화 → 프롬프트/템플릿.

```
baoyu-comic:    source → analysis.md → storyboard.md → prompts/ → images
baoyu-infographic: source → analysis.md → structured-content.md → prompts/ → image
seminar-slides: (수동 분석) → design.md → HTML
novel-writing:  설정 → DESIGN.md → 본문
```

## 설계 제약

### 기존 스킬 파괴 금지

기존 도메인 스킬(seminar-slides, novel-writing, baoyu-comic 등)은 독립적으로 계속 사용 가능. 메타 스킬은 **래퍼**로 작동하며, 하위 스킬을 호출하거나 템플릿을 상속한다.

### 워크플로우 충돌 방지

| 워크플로우 패턴 | 도메인 |
|----------------|--------|
| 템플릿→검증→출력 (선형) | README, Wiki, 블로그, 슬라이드, 인포 |
| 프롬프트→큐→생성→리뷰 (반복) | 이미지, 만화 |
| 설정→절→화→장 (연속 세션) | 소설 |

각 패턴은 도메인 스킬 내에서 처리. 메타 스킬은 라우팅과 공유 엔진 제공만 담당.

### 템플릿 과다 방지

Phase 1에서는 4개 핵심 템플릿으로 시작. 나머지는 파생 또는 기존 스킬 템플릿 상속.

│   ├── tier-generator.py       # 계층화
│   ├── tone-adaptor.py         # 어조 적응

│   ├── template-filler.py      # 템플릿 렌더링

│   ├── tokens.css              # 디자인 토큰
│   ├── matrix.json             # 스타일×레이아웃 매트릭스
