## 설계 진화 타임라인

### Phase 1: 초기 분석 (문서화 시스템)

12개 도메인 스킬 분석 완료. 공유 인프라 5가지 식별:
- 스타일×매트릭스 시스템
- 콘텐츠 분석→구조화 파이프라인  
- 디자인 토큰+색상 팔레트
- 검증 체크리스트
- 템플릿 렌더링 엔진

### Phase 2: 도메인 확장 검토

사용자 요청: "이미지 생성과 소설 작성까지 아우를 수 있을 것 같은데"

**분석 결과**: 단순 통합은 워크플로우 충돌 초래 (선형 vs 반복 vs 연속 세션)
**대안**: 도메인별 메타 스킬 + 통합 라우터

**사용자 피드백**: "진입점 외에 둘 이상의 컨텐츠 작성이 공유할만한 기능이 있는지 검토해줘. 그리고 여기에 포함시키는건 창의력이나 표현력이 필요한 컨텐츠로 제한해"

### Phase 3: 공유 기능 심화 분석

사용자 지적: "창의적인 내용 구성 및 문장 생성 측면에서 공유하는 부분도 있잖아"

**추가 공유 패턴 식별**:
1. 계층화 (Progressive Disclosure) - 모든 도메인 공통 L1→L2→L3 구조
2. 어조/타겟 적응 - 비기술적↔기술적, 7가지 톤, 21가지 스타일
3. 비유 및 아날로지 생성 - 추상→구체 변환
4. 문장 길이 및 밀도 제어 - 도메인별 자수/밀도 규칙
5. 반복 패턴 템플릿화 - 구성 요소 재사용
6. 기술 용어→일반 언어 변환 - 💡Tip 2층 구조
7. 공통 검증 - 문자셋+일관성+분량 3가지

### Phase 4: 교육/설명 문서 포함

사용자 지적: "리드미, 위키, 블로그는 어디에 해당해? 단순히 요약하는게 아니라 흥미롭고 이해에 도움이되는 결과물이 나와야 하잖아"

**재분류**: README, Wiki, 블로그도 "연출 필요" 도메인으로 포함
- 계층화: Overview→QuickStart→Guide (README)
- 비유: "Docker=컨테이너 박스", "RAG=카탈로그" 
- 어조: 기술적↔비기술적 독자
- 템플릿: Quick Start, FAQ, Contributing

### Phase 5: 정체성 정의

**시스템 정체성**: 연출 (연출)
**연출 대상**: 의미 (의미)

## 워크플로우 충돌 분석

| 워크플로우 패턴 | 도메인 | 템플릿 방식 |
|----------------|--------|-------------|
| 템플릿→검증→출력 (선형) | README, Wiki, 블로그, 슬라이드, 인포 | MD/HTML/SVG |
| 프롬프트→큐→생성→리뷰 (반복) | 이미지, 만화 | ComfyUI JSON, 프롬프트 |
| 설정→절→화→장 (연속 세션) | 소설 | MD (화별 연재) |

**해결**: 각 패턴은 도메인 스킬 내에서 처리. 메타 스킬은 라우팅+공유 엔진 제공만 담당.
