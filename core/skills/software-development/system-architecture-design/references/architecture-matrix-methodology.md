# Architecture Matrix Methodology

> JOB-1626에서 도출한 시스템 감사 및 아키텍처 매트릭스 구성 방법론

---

## 도메인 매트릭스 정의

도메인은 **책임 영역**으로 정의, 폴더명이 아닌 역할 기반 분류.

### 필수 필드

| 필드 | 설명 |
|------|------|
| 도메인 | 책임 영역 이름 (JOB, Knowledge, Cron, etc.) |
| 위치 | 실제 파일 시스템 경로 |
| 역할 | 1문장으로 책임 설명 |
| 파일 수 | 정량적 인벤토리 |
| 크기 | 디스크 사용량 |
| 데이터 수명 | Static / Dynamic / Transient |
| SSOT | 단일 진실 근원 파일/경로 |

### 데이터 수명 분류 기준

| 수명 | 변경 빈도 | 예시 |
|------|-----------|------|
| **Static** | 낮음 | config, skills meta, templates, lessons |
| **Dynamic** | 중간 | knowledge wiki, jobs, reports |
| **Transient** | 높음 | state, locks, events, cache, sessions |

### SSOT 식별 패턴

- 설정 파일: `config/*.yaml`, `registry.yaml`
- 상태 파일: `.workflow-state`, `jobs.json`
- 지식: `wiki/index.md`, `catalog.json`
- 교훈: `lessons/index.md`

## 참조 그래프 구성

### 도메인 간 참조 방향

```
┌──────────┐
│  Config   │ ←── 모든 도메인에서 참조
└────┬─────┘
     │
┌────▼────┐    ┌──────────┐    ┌──────────┐
│  Cron    │───→│    JOB    │───→│ Knowledge │
└──────────┘    └────┬─────┘    └──────────┘
                     │
                     ▼
              ┌──────────┐
              │  Backup   │ ←── 모든 도메인 데이터 백업
              └──────────┘
```

### 정량적 참조 분석 (스크립트 기준)

1. 각 스크립트에서 참조하는 디렉토리 grep: `grep -r "path/to/dir" scripts/`
2. 참조 횟수 집계 → 비율 계산
3. Top N 참조 대상 식별

## 중복 분석 (Overlap Detection)

### 겹침 영역 식별 패턴

1. 두 도메인이 동일한 파일/디렉토리를 참조하는지 확인
2. 설정 vs 코드가 분리되어 있는지 확인
3. 상태 파일이 혼합되어 있는지 확인
4. 명칭과 역할이 일치하는지 확인

### 권장 명칭 명확화

| 모호한 명칭 | 명확한 명칭 | 이유 |
|------------|------------|------|
| Deploy | Infra | GPU/ComfyUI는 코드 배포가 아닌 인프라 프로비저닝 |
| Scripts | 도메인별 분류 | 도메인 경계 명확화 |

## 문제점 식별 체크리스트

- [ ] 단일 디렉토리에 50개 이상 파일 무분별하게 나열
- [ ] 동일 데이터의 2개 이상 복사본 존재
- [ ] 잠금 파일, 상태 파일, 임시 결과가 혼재
- [ ] 백업/아카이브 정책 부재
- [ ] 명칭과 실제 역할 불일치
- [ ] 도메인 경계 불명확
- [ ] SSOT 식별 불가

## 심링크 마이그레이션 전략

### 기본 원칙
1. **이동 후 심링크 생성**: `mv file target/ && ln -s target/file old/path/file`
2. **단계적 검증**: 각 단계 후 cron/job/state 테스트
3. **위험도 평가**: 각 작업의 영향도/위험도 평가 후 순서 결정

### Phase 분할 기준
1. **Low risk**: state 파일 분류, 명칭 변경
2. **Medium risk**: scripts 분류, knowledge 통합
3. **High risk**: cron 관련 스크립트 이동, workflow 변경

### 검증 체크리스트
- [ ] 모든 cron 작업 정상 실행
- [ ] workflow-gate.sh JOB 시작/완료 정상
- [ ] 지식 파이프라인 정상 동작
- [ ] 이미지 생성 파이프라인 정상 동작
- [ ] 상태 파일 접근 정상
- [ ] 심링크로 기존 절대 경로 호환 유지

## LLM 친화적 폴더 구조 원칙

1. **루트 레벨 10개 미만**: LLM이 한 번에 파악 가능한 디렉토리 수
2. **층위별 진입점 명확화**: `AGENTS.md` → 도메인별 `README.md` → 구체 파일
3. **심링크 투명성**: 이동 후 심링크로 기존 경로 유지 → 레거션 방지
4. **카테고리 우선**: 파일명을 폴더 분류로 대체 (`cron-runner-*.sh` → `scripts/cron/runner-*.sh`)
