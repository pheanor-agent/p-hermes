# Spec 중심 개발 워크플로우

## 세션 학습 (JOB-1467, 2026-06-03)

### 사용자 핵심 피드백

> "코드 작성까지는 챙겨야 할 것 같은데"
> "LLM이 코드를 직접 관리하기보다는 설계 사양서를 관리하고, 사양서를 기반으로 코드를 작성 및 사양서 기반 작성이 잘 된건지 리뷰하는 방식으로 코드 안정성을 높이기 위한 컨셉이야"
> "코드에 대한 피드백을 주는 경우에도 사양서 기준으로 피드백을 반영하고 리뷰한 후 코드로 작성, 사양서를 기준으로 코드를 리뷰 이런식으로 진행되게 해줘"

### 재정의된 시스템 철학

**이전 오해**: "Spec 템플릿 복사 → 코드 자동 생성"
**수정된 철학**: "Spec 정의 → Spec 기반 코드 리뷰 → Spec 기준 피드백 → 코드 안정성 확보"

```
Spec (단일 진실 근원)
    ↓ (개발자/LLM이 읽어서 구현)
코드
    ↓ (Spec 기반 리뷰)
일관성 검증
    ↓ (Conformance Score)
품질 확보
```

### Spec 기준 피드백 패턴

**나쁜 피드백**: "이 코드 나빠", "여기 수정해"
**Spec 기준 피드백**: "Spec-A001 조건 2: 토큰 유효기간 1시간으로 정의 → 코드에서 30분으로 설정 (일치 X)"

### 템플릿 변경 사항

**requirement.md**, **interface.md**에 "Spec 기반 코드 리뷰" 섹션 추가:

```markdown
## Spec 기반 코드 리뷰

**리뷰자**: 
**리뷰 일시**: 

### 조건 검증
| # | 조건 | 코드 구현 | 상태 |
|---|------|-----------|------|

### 동작 검증
| # | 동작 | 코드 구현 | 상태 |
|---|------|-----------|------|

### 피드백
- Spec-A001 조건 1: ...

### 판정
- [ ] ✅ Spec 기준 완전 충족
- [ ] ⚠️ 일부 미흡
- [ ] ❌ 재구현 필요
```

### Pitfall: 코드 생성기 오해

사용자 피드백: "생성기 버그: 사양서의 condition과 actions를 무시하고 하드코딩된 템플릿 출력"

**원인**: 시스템이 Spec을 코드 생성기로 오인
**수정**: Spec은 명세(documentation)이며, 코드는 개발자가 Spec을 읽어서 수동 구현
**교훈**: 시스템은 Spec을 관리하고, Spec을 기준으로 코드를 리뷰하며, Spec 기준 피드백을 제공
